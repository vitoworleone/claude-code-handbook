# 第九集：上下文压缩与记忆系统

欢迎回到 Claude Code 的深度解析课程。今天我们来聊一个所有 LLM 应用都会遇到的终极问题：**上下文窗口的局限性**。

想象一个场景：你早上九点开始用 Claude Code 做重构任务，到中午对话已经有 150 条消息，你修改了 20 个文件。这时候 Claude 开始忘记你在第 30 条消息里说的「不要用 async/await」。

这就是上下文窗口问题——所有 LLM 都有这个问题，区别在于如何优雅地处理它。

Claude Code 的解决方案是一个精心设计的分层系统：
- **上下文压缩** → 处理当前会话的旧内容
- **CLAUDE.md 记忆** → 处理跨会话的持久知识

## 问题有多严重

Claude 的上下文窗口大约是 **200K Token**。听起来很大对吧？但来做一道数学题：

| 项目 | Token 数 |
|------|----------|
| 系统提示 | 2,000 |
| 工具列表 | 5,000 |
| CLAUDE.md / 记忆 | 1,000 |
| **固定成本合计** | **8,000** |

然后每次对话轮次还要加上 3,000-10,000 Token。到第 20 轮，你已经用了超过 100,000 Token，接近上限。再来几轮，直接报错。

更坏的情况是：即使没有达到硬性上限，Token 数量越大，Claude 对早期内容的**注意力就越稀薄**——不是完全忘记，而是优先级降低，早期的用户偏好会逐渐被忽略。这就是为什么很多用户抱怨「Claude 越用越不听话」。

## 三种压缩策略

| 策略 | 触发方式 | 时机 | 成本 |
|------|----------|------|------|
| **自动压缩** | Token 数接近阈值时自动触发 | 下一次 API 调用前 | 无额外 API 调用 |
| **响应式压缩** | API 返回 `prompt_too_long` 后触发 | 事后补救 | 多一次失败的 API 调用 |
| **手动压缩** | 用户主动发起 `/compact` | 用户控制 | 用户可指定保留重点 |

三种策略的核心机制一样：**生成摘要，替换旧消息**。区别在于预防还是补救，主动还是被动。

### 自动压缩（最核心）

当系统检测到 Token 数快超阈值时，不会等你爆错了再处理，而是在下一次 API 调用之前**悄悄执行压缩**——用户完全无感知。代码层面就是一个简单的 `if` 判断加状态更新。

### 手动压缩

用户可以告诉 Claude：「这次压缩要重点保留哪些内容」。比如你正在讨论 JWT 认证模块，就可以强调重点保留关于认证模块的讨论，摘要会优先保留这部分信息。

## 压缩的完整过程

```
原始消息历史 (100条, 150K Token)
  ↓
① 运行 PreCompact Hooks（工具可注入必须保留的上下文）
  ↓
② 调用 Claude 生成摘要（主要工作/创建的文件/未完成的 Todo）
  ↓
③ 插入 CompactBoundaryMessage（压缩点标记）
  ↓
④ 运行 PostCompact Hooks（工具可追加额外信息）
  ↓
压缩后 (30K Token, 节省 80%)
```

## CompactBoundaryMessage 的设计

压缩边界标记包含四个关键信息：
1. 压缩时间戳
2. 压缩前的 Token 数量
3. 摘要内容
4. 保留了最近多少条原始消息

**为什么要单独设计这个标记？三个原因：**

**1. 调试**：用户抱怨 Claude 忘记了某个约束，开发者可以查看压缩边界，判断这个约束是在压缩前（被摘要化了）还是压缩后（理论上应该记得）。

**2. 重播（Resume）**：恢复会话时不需要重播压缩前的所有历史，从压缩边界开始即可。

**3. 工具集成**：`TaskTool` 会在这个边界上注册 Hook，在压缩时确保当前运行的后台任务信息被注入到摘要里。

## Hook 系统：压缩的灵魂

Hook 系统让工具可以干预压缩过程，而不是让压缩变成一个黑盒操作。

**两种 Hook：**

| Hook | 执行时机 | 作用 |
|------|----------|------|
| **PreCompact Hook** | 生成摘要之前 | 返回「一定要保留的上下文」 |
| **PostCompact Hook** | 压缩之后 | 修改或追加摘要内容 |

### TaskTool 的用法

当你在后台跑测试或构建任务时，`TaskTool` 注册了一个 PreCompact Hook：

1. 压缩前检查有没有正在运行的任务。
2. 如果有，把任务列表格式化后注入到上下文中。
3. 压缩后 Claude 依然记得：「哦，还有三个测试任务在跑，我需要检查结果。」

如果没有这个 Hook，Claude 可能会压缩后就忘记后台任务的存在，去干别的事情了。

**这就是系统设计的精妙之处：压缩不可避免，但可以通过 Hook 最小化信息损失。**

## CLAUDE.md 记忆系统

上下文压缩和 CLAUDE.md 是互补的两极：

| 维度 | 上下文压缩 | CLAUDE.md |
|------|-----------|-----------|
| 处理范围 | 会话内的历史 | 跨会话的持久指示 |
| 信息损耗 | 有损耗（摘要） | 无损（全文保留） |
| 注入方式 | 压缩后替换旧消息 | 每次 API 调用原封不动 |

**核心原则：任何你希望 Claude 永远记住的事情，都应该写进 CLAUDE.md。**

比如：
- 这个项目不使用 async/await
- 所有 API 路由必须加认证中间件
- 测试框架用 Vitest 不是 Jest

压缩后的记忆是有损耗的摘要，但 CLAUDE.md 的内容每次 API 调用都原封不动地送进去。
