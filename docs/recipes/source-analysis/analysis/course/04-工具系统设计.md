# 第四集：工具系统设计

欢迎来到第四课——工具系统设计。

在开始之前，我想问大家一个问题：如果你要设计一个工具调用系统，工具的定义应该包含哪些字段？

DeepFlow 的工具定义是这样的：两行代码，一个函数加一个描述字符串，简单直观。

那 Claude Code 的工具接口有多少个字段呢？**答案是九个。**

包括：执行函数、权限检查、只读标记、破坏型标记、并发安全标记、三个渲染函数、延迟加载标记、搜索提示。

为什么这么复杂？这就是我们今天要解答的问题。Claude Code 的每一个字段背后都是一个真实的工程需求，不是什么过度设计。

## Tool 接口的完整定义

按功能分组，九个字段可以分成七个功能群：

| 功能群 | 字段 | 作用 |
|--------|------|------|
| 身份标识 | `name`、`alias` | 唯一标识工具 |
| 执行核心 | `inputSchema`、`call` | Zod 输入验证 + 实际执行逻辑 |
| 权限管理 | `checkPermissions`、`isReadOnly`、`isDestructive` | 权限检查 + 声明 |
| 并发控制 | `isConcurrencySafe` | 能否和其他工具并行执行 |
| 描述系统 | `description` | 给 LLM 看的工具说明（函数形式，可根据上下文动态生成） |
| 渲染系统 | `renderToolUseMessage`、`renderToolUseProgressMessage`、`renderToolResultMessage` | 给用户看的三个阶段界面 |
| 搜索优化 | `shouldDeferAlwaysLoad`、`searchHint` | 控制工具加载时机 |

## 执行核心：Zod Schema 的三层能力

`inputSchema` 用的是 **Zod**——一个 TypeScript 优先的模式声明库。

**一个定义，三层能力：**

```typescript
const bashSchema = z.object({
  command: z.string().describe('The command to execute'),
  timeout: z.number().optional().describe('Timeout in milliseconds'),
});

// 第一层：TypeScript 类型 → 编译时类型检查
type BashInput = z.infer<typeof bashSchema>;

// 第二层：运行时验证 → 验证 LLM 返回的格式
const parsed = bashSchema.parse(llmOutput);

// 第三层：参数描述 → 自动注入 API Prompt
// .describe() 的内容会被提取，告诉 LLM 这个参数是干什么的
```

另外，有些工具可以开启 **strict 模式**，禁止 LLM 传入 schema 之外的字段——防止 LLM 创造性地传入额外字段来绕过校验。

## 权限模型：声明与检查分离

每个工具有两层权限设计：

**第一层：权限声明**
- `isReadOnly(input)` — 返回 `true` 表示这个工具在给定输入下是只读的，不会修改文件。
- `isDestructive(input)` — 返回 `true` 表示这是个破坏性操作，可能删除或覆盖数据。

**第二层：权限检查**
- `checkPermissions(input, context)` — 真正的权限决策逻辑。查看当前权限模式、用户的允许/拒绝规则，然后决定是否放行。

**为什么要分开？** 因为用途不一样：
- **声明**是给 UI 看的——只读工具可以在界面上显示不同颜色，让用户一眼就知道这个工具不会修改代码。
- **检查**是给系统看的——决定实际能不能执行。

## 三层渲染系统

每个工具有三个渲染函数，对应工具调用的三个阶段：

| 阶段 | 渲染函数 | 示例 |
|------|----------|------|
| 调用开始 | `renderToolUseMessage` | 「正在调用 BashTool: `ls users/ tell project src`」 |
| 执行中 | `renderToolUseProgressMessage` | 「运行中... 已执行 12s，输出 45 行」 |
| 完成 | `renderToolResultMessage` | 语法高亮的结果，可折叠 |

这个设计的本质是：**工具调用的 UX 由工具自己定义**，不是统一的「工具调用盒子」。

- `FileEditTool` 显示 diff
- `BashTool` 显示语法高亮的终端输出
- `WebFetchTool` 显示 URL + 摘要

每种工具的输出都有最适合它的呈现方式。代价是每个新工具需要实现三个渲染函数，增加了开发量，但用户体验完全不同。

## 工具组装管道

工具从代码实现到最终出现在 API Prompt 里，经历四个步骤：

```
GetAllBaseTools → FilterByDenyRules → MergeMcpTools → UniqueByName
```

### 第一步：获取基础工具集

`getAllBaseTools()` 返回所有内置工具：AgentTool、BashTool、GlobTool、GrepTool 等等。注意 `RepoTool` 是条件加载的。

### 第二步：按权限过滤

`filterToolsByDenyRules()` 根据用户的拒绝规则过滤。比如用户 deny 了 Bash，那 BashTool 就会被移除。

### 第三步：合并 MCP 工具

**关键细节：内置工具和 MCP 工具是分开排序的**，不是混在一起排序。为什么？待会解释。

### 第四步：去重

`uniqueBy('name')` 去重。MCP 工具可能和内置工具重名，这时内置工具优先。

## 分开排序的秘密：Prompt Cache 缓存稳定性

为什么内置工具和 MCP 工具要分开排序？

Anthropic API 的 **Prompt Cache** 机制会缓存工具列表的前缀。如果内置工具和 MCP 工具混合排序：

- 用户新增一个 MCP 工具 → 整个工具列表的排序发生变化
- 所有工具的 Prompt Cache 全部失效
- 每次调用都要重新处理完整工具列表

**分开排序后：**
- 内置工具始终排在前面，顺序稳定 → Prompt Cache 始终命中
- MCP 工具排在后面，只有 MCP 部分变化时才会影响

**每次 API 调用节省数千 Token 的重复计算。** 这就是分开排序背后的工程考量。
