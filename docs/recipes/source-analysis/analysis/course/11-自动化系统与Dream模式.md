# 第十一集：自动化系统与 Dream 模式

大家好，欢迎来到第十一集。

上一集我们讲了用户手动写入的持久化记忆（CLAUDE.md），但 Claude Code 还有另一套更神奇的系统——**自动记忆**。

不需要你手动写一个字，Claude 会在每次对话后自动把重要信息提取出来。更特别的是，每隔一段时间，Claude Code 会在后台「做梦」——启动一个隐形的 Claude 实例，把积累的碎片化记忆重新整合成结构化的知识。

这集我们就来看看这套系统是怎么设计的。

## 学习目标

1. **Stop Hooks 管道** — 每轮结束后触发后台任务的框架（总调度器）
2. **Extract Memories** — 每次对话后自动提取记忆（增量写入）
3. **AutoDream** — 后台记忆整合系统（三关门设计）
4. **Away Summary** — 离开一段时间后回来的摘要
5. **RunForkedAgent** — 所有系统共享的底层基础设施

## Stop Hooks 管道

每次 Claude 完成一轮回复（没有工具调用，最终输出完成时），触发 `handleStopHooks` 管道，按顺序执行六个步骤：

| 步骤 | 操作 | 模式 |
|------|------|------|
| 1 | 保存 PromptCache 快照 | 同步（后续后台任务依赖此快照共享缓存） |
| 2 | 分析下一步操作建议 | Fire-and-Forget（不阻塞主线程） |
| 3 | 提取会话记忆（Extract Memories） | Fire-and-Forget |
| 4 | 检查是否触发 Dream | Fire-and-Forget |
| 5 | 用户配置的 Web Hooks | 阻塞（需要等结果） |
| 6 | Swarm 模式下的 Teammate 协调钩子 | 阻塞 |

**注意：非交互模式下步骤 2-4 全部跳过。** 脚本调用不需要自动记忆。

### 为什么是 Stop Hooks 里触发？

每轮 Query 结束时，三个条件同时满足：

1. **消息历史完整** — 包含这轮所有工具调用和结果
2. **PromptCache 刚更新** — Forked Agent 能直接共享这批 Cache
3. **用户等待输入** — 短暂的空闲窗口

这是做后台工作的**黄金时间点**。而且 Fire-and-Forget 的任务不阻塞主线程——用户完全无感知，以为 Claude 在等输入，实际上后台正在默默整理记忆。

## Extract Memories（第一层自动记忆）

每次 Query 结束后，在后台启动一个 **Forked Agent**——一个完全复制了当前 Agent 上下文的 Claude 实例。这个 Forked Agent 会：

1. 分析这轮对话
2. 判断哪些信息值得记住
3. 写入 Memory 目录

**关键设计点：**

| 特性 | 说明 |
|------|------|
| 共享 PromptCache | 复用主 Agent 的工具描述缓存，节省 Token |
| 最多运行 5 轮 | 防止陷入无限循环 |
| 权限受限 | 只能写 Memory 目录，Bash 只读 |
| UI 提示 | 完成后显示 "Saved N memories" |
| 只主 Agent 触发 | 子 Agent 跳过（避免递归） |

### 为什么用 Forked Agent 而不是直接 API 调用？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **直接 API 调用** | 实现简单 | 需维护独立 Prompt + 工具列表；无法共享 PromptCache；需独立权限系统 |
| **Forked Agent** | 自动共享 PromptCache；复用权限系统；实现统一 | 稍复杂 |

**算一笔账：** 主 Agent 的工具描述大约 5,000 Token，已被 API 缓存。Forked Agent 用相同的工具列表，自动命中这个 Cache——每次 Extract Memories 节省 5,000 Token 的费用。

如果每次 API 调用成本是 $0.003/千 Token，每天 100 次 Extract Memories 调用：
- 每天节省：100 × 5K Token × $0.003/K = **$1.5/天**
- 大型团队每天数千次调用 → 节省非常可观

**这是一个以复杂性换成本的经典 Tradeoff。**

## Cursor（游标）机制

如果不加控制，每次对话结束都处理全部历史消息 → 重复处理、重复写入、效率低下。

所以 Extract Memories 维护了一个**游标**（`lastMemoryMessageId`）：每次只处理游标之后的新消息。

```
第一轮：处理消息 1-10 → 游标移动到消息 10 的 UID
第二轮：只处理消息 11-15 → 游标移动到消息 15
第三轮：只处理消息 16-20 → 游标继续推进
```

**关键容错设计：** 如果上下文压缩删除了游标指向的消息怎么办？

答案是：**回退为计数全部消息，而不是返回零。** 因为如果返回零，提取器会认为没有新消息，永久跳过——这是一个严重的 Bug。回退到全部计数，至少保证功能不会永久失效。

### 另外两个保护机制

**1. 互斥检查**

如果主 Agent 在这轮对话中已经直接写过 Memory 文件，Extract Memories 跳过本轮——避免重复写入和潜在冲突。

**2. 重叠保护**

两个 Extract Memories 任务不会同时运行。用简单的标记位防止并发执行。

## AutoDream（后台记忆整合）

这是第二层自动记忆系统，也是最精妙的部分。

### 三关门设计

Dream 的触发有三个条件（三门）：

**第一门：时间门**
- 距离上次 Dream 是否超过阈值（默认 N 小时）
- 积累的新记忆是否足够多

**第二门：空闲门**
- 用户当前是否空闲（没有活跃对话）
- 不能在用户正忙时启动 Dream

**第三门：资源门**
- 系统资源是否充足
- API 配额是否够用

只有三门同时通过，Dream 才会启动。

### Dream 做了什么

1. 启动一个**隐形的 Claude 实例**（Forked Agent）
2. 读取所有碎片化的 Memory 文件
3. 重新整合：去重、合并相关记忆、提取高层知识
4. 将整合后的结构化知识写回 Memory 目录
5. 删除已被整合的碎片记忆

这就像人脑在睡眠中整合白天学到的知识——**从碎片到结构，从短期到长期。**

## Away Summary

当你离开一段时间再回来，Claude 会给你一个摘要：「你离开期间发生了什么」。

实际上不是 Claude 一直在监控——而是通过 Stop Hooks 和 Dream 系统积累的状态，在你回来时生成一个总结。

## 总结

整个自动记忆系统的设计哲学是：

- **增量提取**（Extract Memories）：每次对话后提取 → 游标防重复
- **定期整合**（AutoDream）：后台 Dream 合并碎片 → 三关门控制触发
- **基础设施共享**（Forked Agent）：所有后台任务共享 PromptCache → 降成本
- **用户无感知**（Fire-and-Forget）：不阻塞主线程 → 体验流畅

这套系统让 Claude Code 从「每次对话都是新的」变成了「越用越懂你」。
