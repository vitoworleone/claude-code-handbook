# 第二集：启动流程 — 并行化 IO 与延迟加载

同学们好，欢迎来到第二集。

今天我们来聊一个很少被认真对待的话题：**CLI 工具的启动流程**。

我先问你一个问题：你的 `main.js` 或 `main.py` 第一行通常是什么？

大多数人的答案是 `import xxx` 或者 `const x = require('xxx')`。

那 Claude Code 的 `main.tsx` 第一行是什么？

**不是 import——是一个性能打点。**

```typescript
profileCheckpoint('mainStart');
```

第十行是两个异步 IO 操作：**Keychain 读取**和 **MDM 配置获取**。在你读完所有 import 之前就触发了。

这集讲的就是：为什么 Claude Code 的启动流程这么做，以及这个设计模式背后的工程哲学。

## 时间线对比

先建立直觉。不好的写法是这样的：

```
0ms    开始加载所有模块
500ms  模块加载完成，开始读取 Keychain
700ms  Keychain 读取完成
800ms  用户才能输入
```

用户总共等了 800ms。

Claude Code 的写法聪明在哪里？

```
0ms    同时做两件事：
       ├─ 开始 import 模块（CPU 忙）
       └─ 后台触发 Keychain 读取（IO 等待）
500ms  import 完成，Keychain 结果已在内存中
600ms  用户就能输入
```

节省了整整 200ms。

关键动作是：**import 过程本身需要时间，CPU 在执行 JavaScript，但 IO 可以并行进行。把重量级 IO 操作提前触发，让它们在 CPU 忙着加载模块时在后台完成——这就是并行化 IO 等待的经典模式。**

## 代码实现

`main.tsx` 第 9 行是性能打点，然后第 10-11 行并行触发了两个重量级操作：

1. **`startMdmRead`** — 读取企业 MDM 配置，决定某些策略限制。
2. **`startKeychainPrefetch`** — 从 macOS Keychain 读取 API Key。

这两个操作都是系统调用，耗时大约 100-200ms。

如果等到需要用时才发起，用户在第一次 API 调用时会额外等待 200ms。但通过在 import 评估期间就触发，这 200ms 和后续的模块加载时间重叠了——用户完全感知不到。

## 四阶段架构

`main.tsx` 有 4683 行，可以划分为清晰的四个阶段：

| 阶段 | 行数范围 | 职责 |
|------|----------|------|
| Phase 0 | 前 20 行 | 并行触发重量级 IO |
| Phase 1 | 21-200 行 | 模块设置：延迟 require、特性门控、条件导入 |
| Phase 2 | 201-3700 行 | CLI 解析：参数解析、配置加载、认证检查、权限加载 |
| Phase 3 | 3700 行以后 | UI 启动：`launchRender` 渲染 TUI，或进入 Headless/SDK 模式 |

这个分层不是随意划分的，它解决了一个很实际的问题——**每个阶段的失败有不同含义，错误上下文精确**：

- **Phase 0 失败**：几乎不可能，是操作系统级别的问题。
- **Phase 1 失败**：说明某个功能模块损坏，可以给出模块名。
- **Phase 2 失败**：参数格式错误，直接打印帮助文档——这时候甚至不需要启动 TUI，纯文本就够了。
- **Phase 3 失败**：TUI 初始化出错，fallback 到纯文本输出。

这比一个 `main` 函数里 try-catch 所有错误要好得多。分阶段让每阶段的错误上下文清晰，用户的错误提示也更友好。

## 循环依赖解决方案

在 Claude Code 里，依赖链是这样的：

```
main.tsx → tools → AgentTool → coordinator → main.tsx  ← 循环了！
```

Node.js 和 Bun 遇到循环依赖时，被循环导入的模块可能是半初始化状态，导致 `undefined` 错误。

Claude Code 的解决方案是：**懒加载工厂函数**。

```typescript
// 先定义一个函数类型
type AgentToolFactory = () => AgentTool;

// 通过 ifFeature 条件判断，把 require 包在一个函数里返回
const getAgentTool: AgentToolFactory = () => require('./agent/agentTool');
```

这样，这个函数只在被调用时才执行 `require`。此时所有模块都已经完全初始化，循环依赖不再是问题。

同时，`ifFeature` 门控确保 OSS 构建里这行 `require` 完全不存在——编译器在 Dead Code Elimination 阶段直接删除了。

## Bun 的特性门控 vs 传统 Feature Flag

这里有一个本质区别：

**传统 Feature Flag（运行时判断）：**
```typescript
if (process.env.FEATURE_X) {
  import('./enterpriseModule');
}
```
问题：这段代码永远在 Bundle 里，只是运行时不一定执行。用户下载了你的软件，企业功能的代码全在里面。

**Bun 的特性门控（构建时判断）：**
```typescript
ifFeature('ENTERPRISE', () => import('./enterpriseModule'));
```
如果 `ENTERPRISE` 为 `false`，整个 `if` 块被删除，连 `import` 语句都不会出现在 Bundle 里。这是**死代码消除**，不是运行时隐藏。

OSS 用户下载的包里根本就没有企业功能的代码。

Claude Code 用这个机制维护了三个构建变体：

| 版本 | 特性 |
|------|------|
| OSS | 基础工具集 |
| Enterprise | + Coordinator Mode、Keros |
| Internal | + 所有功能、Coose |

同一个代码库，通过特性门控控制编译结果，不需要维护三个代码分支。

## 权衡

当然，有所得必有所失：

1. 必须在构建时确定特性集合，不能运行时动态切换。
2. 代码里散布着 `ifFeature` 判断，阅读时需要关注当前构建上下文。
3. 新人需要理解：我写的这段代码在某些构建里根本不存在。

但换来的收益是：**更小的包体积、更快的启动速度、企业代码不会泄露到 OSS 版本**。对 Claude Code 来说，这笔交易值得。
