# 第六集：状态管理架构 — 为什么不用 Redux/Zustand？

欢迎来到第六集——状态管理架构。

React 的状态管理通常有三种方案：

1. `useState` + `useContext` — React 内置方案
2. Redux — 经典外部状态管理
3. Zustand、Jotai、Recoil — 现代轻量方案

那 Claude Code 用了哪个？

**答案是：都没有。** 它自己实现了一个 Store，然后用 React 18 的 `useSyncExternalStore` 来订阅。

为什么？这一集告诉你。

## 终端渲染的特殊挑战

要理解 Claude Code 为什么自己做状态管理，先看一个关键问题：

在浏览器 React 里，Context 更新会触发消费该 Context 的组件重渲染。React 会在微任务里批量合并重渲染，通常性能是可以接受的。

**但是在 Ink 终端渲染环境里，情况完全不同：**

每次重渲染意味着重新计算整个终端界面的布局——这涉及到 Yoga 布局引擎（就是 Yoga 那套东西）。更糟糕的是，Ink 的批量合并机制在某些版本里不如浏览器可靠。

结果就是：**频繁的 Context 更新会导致终端闪烁，用户会看到文字跳动。**

特别是对话消息——它是流式的，每个 Token 到来都要更新状态。如果消息状态放在 Context 里，每个 Token 都触发全树重渲染，那个闪烁会非常明显。

**解决方案很直接：把状态放到 React 生命周期之外，组件只订阅自己关心的部分。**

## Claude Code 的 Store 实现

Store 实现非常简洁，只有 **30 行代码**：

```typescript
// 用闭包变量保存状态
let state = initialState;
// 用 Set 存储所有监听器
const listeners = new Set<Listener>();

function getState() { return state; }

function setState(update: (s: State) => State) {
  const newState = update(state);
  if (Object.is(state, newState)) return; // 没变就不通知
  state = newState;
  listeners.forEach(l => l()); // 通知所有订阅者
}

function subscribe(listener: Listener) {
  listeners.add(listener);
  return () => listeners.delete(listener); // 返回取消订阅函数
}
```

### 三个关键细节

**1. `Object.is` 浅相等检查**

强制调用方必须使用**不可变更新模式**。如果你直接修改 State（`state.field = value`），`update` 返回的还是同一个对象引用，`Object.is` 会返回 `true`，直接 return，不会通知任何订阅者——这就是一个隐形的 Bug。

所以你必须用扩展运算符 `{ ...state, field: newValue }` 产生一个新对象。

**2. 用 Set 而不是数组来存储监听器**

Set 自动去重——同一个 Listener 不管你注册多少次，实际只存一份。避免意外注册两次导致重复触发。

**3. subscribe 返回取消订阅函数**

组件卸载时调用这个函数，把监听器从 Set 里移除，不会造成内存泄漏。这是 React Hooks 的标准模式。

## useSyncExternalStore + Selector：精确订阅

React 18 新增了一个 Hook 叫 `useSyncExternalStore`，专门用于订阅 React 管理之外的**外部状态**。

Claude Code 封装了一个 `useAppState`：

```typescript
function useAppState<T>(select: (state: AppState) => T): T {
  return useSyncExternalStore(
    store.subscribe,
    () => select(store.getState())
  );
}

// 使用示例
const model = useAppState(s => s.mainLoopModel);
const taskCount = useAppState(s => Object.keys(s.tasks).length);
```

`useSyncExternalStore` 帮你处理订阅逻辑。当 Store 通知变化时，它调用你的 Selector，然后用 `Object.is` 对比结果——如果结果没变，组件就不重渲染。

**这就是精确订阅的秘密。**

## Context vs Selector 对比

用 Context 的话：一旦 `AppState` 变化，**所有消费这个 Context 的组件都会重渲染**，不管它们实际上有没有用到变化的字段。很多组件在做无效渲染。

用 `useSyncExternalStore` + Selector：
- Store 通知变化
- React 调用所有组件的 Selector 函数
- 对比结果
- **只有真正用到变化字段的组件才会重渲染**

举个数字感受一下：Claude Code 有 50 多个组件在消费 AppState。每个 Token 到来都要更新状态——这是高频事件。

- 不用 Selector：每次 Token 变化都触发 50+ 次重渲染
- 用 Selector：每次可能只有 2-3 个组件真正需要更新

这就是精确订阅的威力。

## DeepImmutable：编译时保障

刚才说 Store 的 `Object.is` 比较会强制不可变更新，但这是**运行时**检查。有没有更早的保障？

有——`AppState` 用了一个 `DeepImmutable` 包装类型：

```typescript
type DeepImmutable<T> = {
  readonly [K in keyof T]: DeepImmutable<T[K]>;
};
```

这是 TypeScript 的条件类型，递归地把所有字段加上 `readonly`。效果就是：

- 如果你试图直接修改 State → TypeScript 编译器直接报错：「这个字段是只读的，不能赋值」
- 你必须用不可变更新：`{ ...state, settings: { ...state.settings, model: 'new' } }`

在编译阶段就把错误堵住，比等到运行时才发现问题要安全得多。

> 不过要注意：TypeScript 的 `readonly` 只是类型系统约束，运行时 JavaScript 并不强制执行。但结合 Store 里的 `Object.is` 检查，形成**双重保障**：编译时拦截 + 运行时验证。
