## 一、开场与嘉宾介绍

**主持人**：我是可烈。最近大家都在讨论 Agent Harness——Agent 的上限是由 Harness 去决定的。可是当我们聊到 Harness 的时候，我们到底在聊什么？

不久之前，Claude Code 的源代码被泄漏，许多 Agent Harness 的关键模块也被完整地呈现了出来，成为了非常好的一个教学样本。

今天我们请到的嘉宾是心路。心路是某 AI 公司的创始人，他们做了一个开源教程——Learn Claude Code，在 GitHub 上已经有超过 5 万颗星。

**心路**：Hello，感谢可烈老师邀请。我们这期播客的目标是希望能够让大家感觉到 Harness 不再是一个神秘的或者玄乎的词，然后可以让大家明白它其实就是软件工程的一些方法。那我们就还是从框架化的方式来开始。

**心路**：我现在在做的是给 AI 开发者提供的 AI 工具链。再一句话介绍我们刚才提到的开源教程——Learn Claude Code 就是我们认为最好的 Harness，所以我们做的教程就是通过对它设计模式的解析，让大家理解 Harness 的核心思想。

## 二、什么是 Agent Harness

**主持人**：如果今天你要用一句话向一个完全没有听说过 Harness 的人介绍 Harness，你要怎么说？

**心路**：我会倾向于把模型比喻为一个聪明的大脑——但是它没有身体和手脚的情况下，它只能思考，没办法行动。Harness 就是模型的身体和手脚。

**主持人**：「Agent 的上限是来自 Harness 的设计」——你认可这句话吗？

**心路**：我对这句话赞成一半吧。

首先，AI Native 的能力上限提升肯定还是模型层面的发展，因为模型越聪明肯定它越好。我们看到现在其实大部分模型的能力也都比较够用了。把模型比作人的话，我们智商在 120 到 130 之间。但是我们可以通过去健身、学舞蹈、练习武术、穿上机甲来做业务，让我们自己更加强大。

这个比喻还是有意思的——**机甲就是 Harness**。我觉得它极大地扩充了我们的能力。

## 三、从 Prompt Engineering 到 Agent Harness 的范式转移

**主持人**：我们一开始提到 Learn Claude Code 教程在 GitHub 上有超过 5 万颗星。大概是什么时间点你开始写这个教程？

**心路**：其实应该有 9 个月前了。

**主持人**：当时是出于什么考虑想写这么一个教程？

**心路**：我们当时的想法就是：Claude Code 是非常强大的。你只要把 Claude Code 套一个网页，其实应该会得到一个非常强大的 AI Native 产品。所以我们就会做给 Claude Code 用的工具链。

当时很多人可能还停留在 Prompt Engineering 的阶段——加上很多 Prompt 节点，然后自己可以流转控制，成本又低又可控。所以我觉得这是一个思想的范式转换。

**主持人**：那你会认为像 LangChain、LangGraph 这样构建 Agent 的框架已经彻底过时了吗？

**心路**：关于 Prompt 提示词链、做流转和控制那一套方法，可能在未来的 AI Native 开发过程中越来越不使用了。基于 Claude Code 这种 AI Native 的方法和思想去做事情——包括 Best Practice 这种范式——会越来越清晰，会被个人和企业采用。

---

## 四、Claude Code 源码泄漏与 Learn Claude Code 教程

**主持人**：在我们录播客的前几天，Claude 正好推出了 Claude Manager Agent，也就是把他自己做 Agent Harness 的这一套开放出来给大家用了。既然现在比如说我要开发一个 AI 产品，我就可以直接用 Claude Manager Agent，那大家还有必要自己去搭 Harness 吗？甚至还有必要去了解 Harness 的细节吗？

**心路**：我认为如果你不了解的话，你做出来的产品缺乏灵魂和缺乏可拓展的空间。

但这个就像云服务器——云服务刚出来的时候，可能大家也会说不能直接就用它的一批 API，我们也要去搞明白这个服务器的方方面面。但到后面你发现，基本上 99% 的工程场景下，我们根本就不需要自己搭服务器。

所以我想，这在 Agent Harness 这个层面是不是也一样呢？我们现在算这期播客还没开始聊——我们要讨论 Harness、要让大家学习 Harness。但会不会有一天，其实根本就不需要学了，大家直接用 Claude 的现成服务就够了？

**主持人**：如果从整个推演角度来看，肯定未来会有那么一天。就像大家今天开发很多全栈的 Web 应用的展示框架——但你不会知道在这里面的底层细节。

**心路**：是的。从长期来看，服务化的方向是确定的。但是现在还处于非常早期的阶段，很多东西还没有定型。我觉得对于 AI 开发者来说，理解 Harness 的运作原理，就像全栈开发者理解 HTTP 协议一样——你可能不需要自己实现，但理解它让你能做出更好的产品判断。

## 五、Claude Manager Agent——还需要自己搭 Harness 吗

**主持人**：那说到 Claude Manager Agent 推出这件事，它说明了什么趋势？

**心路**：Claude Manager Agent 的推出说明服务化是确定的。但是——就像 OpenAI 推出 API 之后，不代表大家就不需要学习 NLP 了。我觉得对 AI Native 开发者而言，学习 Harness 这件事本质上是在学习「模型如何跟工具交互」「如何管理状态」「如何做上下文管理」——这些是基础能力，不会因为服务化就消失。

## 六、Agent 开发的共识与非共识

**主持人**：在今天我们聊到 Harness 的时候，你觉得有哪些是共识、有哪些是非共识？

**心路**：共识可能就是说 **Best Practice**（最佳实践）是一个共识——大家现在都在做 Self-Learning。给 Agent 各种各样的工具是第一重要的。

**主持人**：可以列举一下现在最主要的工具有哪些吗？

**心路**：我觉得最主要的工具是关于文件系统的那些工具——增删读写搜索。这是不论在做编程任务还是做其他任务基本上都会需要的工具。

第二层我认为就是 RAG / Search 这一层——这是给 Agent 关于原始知识系统的一个操作能力的封装层。第三层的就比较有意思——一个语言解释器层。

我觉得大部分情况下你配好这些工具，应该就可以满足我们 95% 以上的任务需求。

---

## 七、工具层设计

**主持人**：听起来配工具这一层很简单，但是这一层有你认为比较容易踩的坑吗？

**心路**：很多时候它跟我们的权限——包括 Agent 的角色——是相关的。比如说我的 Agent 就是负责对代码库做一遍 Review，那么这个时候我应该只给它配只读的工具——比如说所有关于写的命令，包括删改的能力，包括有一些你不能让它乱操作的权限。

如果你配给它以后，你是信任它还是防着它？这些都会有很多需要仔细斟酌的地方。所以 **Agent 和工具链的设计要紧密地跟 Agent 的角色去关联和调试**。它对整个系统的操作能力决定了你的 Agent 能做什么、不能做什么。

---

## 八、记忆与上下文层

**主持人**：我们刚才说到第二层——上下文与状态层、环境层的时候，Memory（记忆）是非常重要的。现在也可以看到有不同的记忆方案——不管是模型厂商原生的一些记忆功能，也有第三方开源的插件提供的记忆解决方案。可不可以给大家讲一讲现在你认为 Agent Harness 的 Memory 这一层有哪些主要的解决方案？

**心路**：Memory 这一层的发展目前还处于一个很早的阶段。我把它粗略地分为三类：**规则式的、半规则式的、和完全模型驱动式的**。

目前在落地使用上来看，相对规则式的和半规则式的为多。

| 类型 | 描述 | 代表方案 |
|------|------|---------|
| **规则式** | 基于知识图谱和向量搜索，结构化信息的标识/关联/建索引/扩检索 | 传统 RAG 方案 |
| **半规则式** | 底层 Unix 文件系统 + Markdown 文件，通过文件夹空间关系组织，Agent 驱动更新 | Claude Code / Cursor |
| **模型驱动式** | 完全由模型自主决定何时存储、如何检索 | 还在早期，落地较少 |

**心路**：我个人其实不太喜欢完全规则式的做法。我更喜欢的是半规则式的——比如底层是一个 Unix 文件系统，我就是通过大量 Markdown 文件的方式来存储重要信息。Claude Code 也是这样做的，Cursor 里面也是这样做的。它可以通过一些类似空间关系——比如说文件夹的方式。底层的文件文件夹结构人也很容易懂，大模型也很容易懂。

它的更新也是半规则式的——通过 Agent 来做一个更新，而不是完全替换掉一个结构。你把这个流程的规则告诉 Agent，但是整个分析和过程是 Agent 来跑的。比如说我要找什么信息，完全是一个后台的 Agent 可能用一个更便宜的小模型去跑一遍。

举个例子：隔几天让它跑一下最近几个 session 的对话，提取有用信息，对过去的记忆做整合和合并。

**主持人**：包括 Memory 这一块有一个开源项目叫 Memory，也非常值得一看。

**心路**：对，Memory 是一个开源项目。它其实非常趋向于用 Markdown + Unix 文件系统 + Agent 去驱动维护、更新和查询的整个方式。Claude Code 里边也是这样设计的。感兴趣的话大家可以去看一下 Learn Claude Code 开源仓库里面对 Memory 这个项目的分析。

**主持人**：Y Combinator 的创始人也在推他自己的个人记忆开源项目。但我觉得大家有点过于混淆所谓的 Memory——机会在于中间的那些边界。

**心路**：所以早做这件事的是 Claude Code——在去年底的时候发了一个版本，里面有一个叫 **Insight** 的功能。它可以分析你最近大概一个月左右的对话，然后分析任务完成情况——它犯了什么错，就给你生成一个总结报告。你就可以知道它的工作状态。

所以我觉得目前关于 Self-Learning 这部分，最成熟的实践就是在 Claude Code 里面的 Insight 那个功能。还有 Memory 这个项目也非常火——它更像是一个 Long-term Memory。

## 九、Harness 的好坏标准

**主持人**：当我们说到一个 Harness 做得好或者做得不好的时候，什么叫好、什么叫不好？

**心路**：不好的 Harness 就是——它会随便去做一些中间的裁剪，去重造一些前面的轮子，或者说随意地修改 Prompt 的实现——这会导致 Prompt 开始失效。

过去一到两年，我发现很多开发者非常关心上下文管理的问题。其实最好的上下文管理就是**不要做管理**——因为你做管理真的会导致它开始失效。只要做了就应该重新算一遍。

**主持人**：所以不好的做法就是跟模型的运行不自洽的。

**心路**：对。好的 Harness 的设计是跟模型的运行是自洽的，跟模型未来能力的进步是正交的。

比如说有一些 Harness 设计——这一步让模型干什么、下一步让模型干什么——但是因为当时的模型能力非常弱，只能做简单的问答，或者做一些简单的功能就已经很厉害了。那这个时候你必须加大量的人工 Craft 的做法。但是这个东西随着模型的进步，就必须拆掉了，要不然就是限制了模型的能力。

**好的 Harness 设计**要符合模型本身在运行上的设计，要符合模型未来能力的进步的设计。

**主持人**：不同的 Harness 在完成同样任务的时候，即便表现相同，好的 Harness 可能消耗的 Token 数量也会不一样，或者完成一个任务的时间也不一样。这背后是不是也是判断 Harness 好坏的一个标准？

**心路**：是的。首先我觉得现在还处于 AI Native 模型发展的早期。OpenAI 在去年率先转型去做推理模型这一代，才有我们今天的 AI Native 模型。其他包括 OpenAI 在内的厂商，是从 2025 年下半年——其实已经落后了半年以后，才开始从传统模型去追推理模型这一代的方向。

所以到目前为止，我认为还是处于 AI Native 模型的早期——其实很多东西都还没有定型。我觉得可能还不必要过多、过快地讨论所谓的「赛道论」。当然这也非常重要，但我觉得对我们使用者来说，你就对标最好的模型——我们的 Harness 也好，我们做的性能也好，都只需要对标最好的模型以及次好的模型，去构建和做产品、做工具。

**主持人**：最好的模型是哪个？次好的呢？

**心路**：最好的可能是 OpenAI，次好的可能像 Claude、还有 Minimax / 智谱他们最新一代的模型，都算是紧跟着最好一代的。

## 十、模型能力进步对 Harness 的影响

**主持人**：如果未来 AI Native 模型因为变得特别强之后，Harness 之间的水平差异还重要吗？假如我们说时间已经到了两三年以后。

**心路**：理论上来讲，不应该有那么多的 Harness。首先好的 Harness 要跟模型的运行原理是贴合的，而且是自洽的。其次好的 Harness 应该是跟模型的进步方向是正交的——模型越强，模型加上你的 Harness 构建的一套 AI 系统，它的能力应该是越强的。

**主持人**：那 Harness 的胜负手可能不在于 Harness 本身了，可能更多在于跟模型的结合程度。

**心路**：对。如果一个 Harness 今天的设计是跟模型的运行规律不贴合、跟它的进步方向不贴合——它只是在某个特定模型版本上做得很好——那随着模型的升级，它一定会被淘汰。

---

## 十一、未来方向：LoRA / Unified Harness / 个性化训练

**主持人**：像 OpenAI GPT-4.0 那个版本——我觉得那已经非常厉害了。所以我在想，类似「插卡」的这种方案可能就很需要——就是把答案的卡片放在一起做高速互联，然后通过自由化的设计，让我们可以用更低的成本和自然的开销去做 PEFT 的训练。然后就是这种高效、低成本的训练——每个人都会得到自己的个性化训练好的模型。

**心路**：其实今天光训练也还不太够，你还要推理。比如我推理的时候，我可能需要一个基础模型加上我的个性化模型参考那一部分。

那今天一个瓶颈就是 LoRA——我能不能说我的 API 请求过去的时候，我的 Header 信息中带有我的 LoRA 的信息？那我可以 reference，然后我还是向发一个请求给大模型，但是它会调用我的 LoRA 信息，可以在推理阶段做一个个性化推理。这样推理的成本可能也很低——因为我关于 LoRA 上去推理，那我可能多支付 5% 的成本，其实也不高。但我获得了一个可以修改模型参数层的一个更加个性化的体验。

所以我会非常关注这个方向的进展。

**主持人**：现在有谁在做这个方向吗？

**心路**：有一些项目在做，但我觉得做的还不太符合预期。因为它其实跟我们的工具链方向也很互补。

我们工具链有一个数据 Header 信息——你运行了半个月，上面可能会存有大量的 Bad Case，你要针对性做分析，做点对点调优。但是可能说，你没有钱、没有卡去修改这个模型——你只能做上下文层面的调优，你没有办法去说「OK，我真的去修改这个模型的参数」，去得到一个我的个性化模型——它可能越来越适配我的应用系统，越来越适配我的 CRM 系统，适配我的应用场景。

**主持人**：你觉得未来你们在做的这个 Unified Harness 这一步，会催生很多公司吗？

**心路**：我觉得会是一个比较激烈的赛道。我们认为未来 Unified Harness 不需要所谓的产品形态多样性。

因为它首先要跟模型的运行是自洽的，其次要跟模型的未来进步方向是自洽的。所以从这两个角度来看——如果同时要自洽这两个点，那么大概率它的结构设计上不会存在很多派系。

应该最多就是：比如说我们比较赞同 Unix + Shared Memory 这种——提供一个 Unix 的 capability 这个派系，然后会有虚拟化的机制，做到数据结构上，然后应用层也是 Shared，然后原来的模块怎么在多个 Sandbox 层面做分配。

但我觉得可能有些其他派系——比如做了以后，它不认这个方向，然后去做严格的规范化控制——可能会有另外的派系。但可能派系不会太多。

**主持人**：那是什么原因你预测到这个赛道的竞争会很激烈？

**心路**：最后其实可能两年之后，没必要等到都没了——还是要去做这个事情的。我们不是最近才进入这个赛道，我们是过去做将近一年的事情了。现在我们只是在演进——去做我们的二代产品，把这个事情做得更加自洽和更加站在终局上来考虑整套事情的做法。

## 十二、创业愿景与结尾

**心路**：我自己创业的初心——每个公司会有自己的使命。我们的使命是「加速世界升级」。

就是说，这件事应该是当前技术条件下对世界升级的最大的一条加速——就像建造更好的火箭或者更好的核聚变手段是对世界升级的最大加速一样。

但当前我们看到的是：如何让 AI 这件事成为社会基础设施——尤其是让「开放 AI」这件事成为整个社会的一种基础设施。然后，未来社会可能让全球有比今天多几百倍的 AI 应用在这个地球上。你怎么去支撑它们？支撑它们自己去开发 AI 的创新应用？

这是一个非常巨大的世界。支撑它们的硬件和工具应该到底是什么样子的？我们其实在思考和实践这样的命题。

所以就回到我们的 Slowband「加速世界升级」——这是我们的做事情的方向和做这件事的初心。
