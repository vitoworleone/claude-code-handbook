一、OpenClaw 与 Claude Code 的架构关系

1.1 核心关系

OpenClaw 的底层决策架构本质上与 Claude Code（CC）相同——都是基于工具调用的循环（tool-calling loop）。两者的区别在于：

  • Claude Code：核心的决策引擎，提供底层 query loop
  • OpenClaw：在 CC 架构之上增加网关层 + 交互层 + 路由层，路由层可将请求分发到 Telegram、微信、飞书等多个渠道

1.2 OpenClaw 的问题

  • 底层架构不可控（如频繁删除文件）
  • 代码量巨大（100 万+ 行），其中大量为垃圾/废弃代码
  • S5 课程策略：借鉴 OpenClaw 的架构逻辑，在 CC 架构之上自建可控的交互框架，而非直接使用 OpenClaw

3.2 三层工具加载机制（核心重点）

第一层：核心工具（6 个）—— 启动时立即加载

工具 功能
bash 执行 Shell 命令
read 读取文件
edit 编辑文件
write 写入文件
glob 文件模式匹配
grep 内容搜索

特点：零外部依赖，启动时直接注册，工具的 schema（描述 + input schema）直接注入 system prompt。

第二层：扩展工具（+11 个）—— 启动后延迟加载

包括任务管理工具、网络工具（HTTPX）、开发工具等。

加载策略：

  • 启动时只注入工具名和描述（schema），不加载依赖
  • 系统启动后异步 import 依赖（HTTPX、Scalito 等）
  • 目的：避免循环依赖，防止启动过慢

第三层：Agent Teams 工具（+3 个）—— 运行时临时构建

触发 Agent Teams 时动态创建的工具：

工具 功能
team_create 创建子 Agent 团队
team_delete 删除团队
send_message 团队间通信

特点（二次布线）：

  • 临时创建，不在启动时注册
  • 创建时需要重新组装上下文（因为工具列表变了）
  • 依赖在运行时才注入（因为依赖取决于当时的状态）
  • 只有触发了 Agent Teams 功能，这些工具才会被创建

三层对比总结

层级 数量 加载时机 依赖加载方式 Schema 注入
第一层 6 启动时 已就绪，随时可用 直接注入 system prompt
第二层 11 启动后 异步延迟 import 先注入 schema，再加载依赖
第三层 3+ 运行时触发 临时注入 + 重建上下文 触发时才构建

总计：6 → 17 → 20+，采用渐进式分步加载。

四、工具调度全流程

4.1 五阶段流程

阶段 1：注册工具
  → 读取 tool_list 字典文件，获取所有已注册工具

阶段 2：生成工具描述清单
  → 将所有工具的 description 和 input schema 组装到上下文中
  → 本质：将 tools schema 注入 system prompt
  → 模型通过 system prompt 知道"有哪些工具可用，每个工具做什么"

阶段 3：模型作出决策（Tool Choice）
  → 用户提问 → 模型根据 system prompt 选择合适的工具
  → 例如：读取文件用 read 而非 cat，查 git 不用 git 命令而用特定工具
  → 输出：文本块 + tool_call 块

阶段 4：执行工具调用
  → 解析 tool_call → 执行对应工具函数 → 获取结果

阶段 5：结果返回模型
  → 工具执行结果作为新的 message 追加到对话中
  → 模型根据结果决定下一步（继续调用工具 or 输出最终答案）

4.2 关键原理

工具调用的本质是 Schema-Driven Prompt Engineering：

  • tools=[tool_schema_1, tool_schema_2, ...] → 最终注入到 system prompt
  • 模型看不到工具的实现代码，只看到工具的描述（description）和参数定义（input schema）
  • 模型通过语义匹配决定"该用哪个工具"

五、Agent Teams 协作机制

5.1 什么是 Agent Teams

Agent Teams 允许在主 Agent 运行过程中动态创建子 Agent 团队，实现多 Agent 协作。

5.2 二次布线（Secondary Tool Wiring）

当 Agent Teams 被触发时：

  1. 创建新工具：临时生成 team_create、team_delete、send_message 等
  2. 重建上下文：因为工具列表变了，需要重新组装 system prompt
  3. 注入运行时依赖：依赖在创建那一刻才确定（可能依赖之前的返回结果）

5.3 子 Agent 的上下文组装

创建子 Agent 时：

  • 需要重新组装上下文（tool list 不同）
  • 子 Agent 有独立的工具集
  • 子 Agent 的结果通过 send_message 回传给主 Agent

术语 含义
Query Loop CC 的核心循环：接收输入 → 模型决策 → 工具调用 → 结果返回 → 循环
