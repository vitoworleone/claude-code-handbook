# 第十集：三大系统对比 — 架构选择的工程逻辑

同学们好，欢迎来到最后一集。

我们研究了三个系统：

| 系统 | 定位 | 技术栈 |
|------|------|--------|
| **NanoClaw** | 个人消息助手 | Node.js + Claude Agent SDK + 消息渠道 |
| **DeepFlow** | 研究自动化平台 | Python + LangGraph + Web API |
| **Claude Code** | 编程助手 CLI | TypeScript + Ink UI + 工具系统 |

这三个系统都在做同一件事：**让 Claude 执行工具完成任务**。但它们的架构选择几乎处处不同。为什么？

**这不是工程师的品位差异，而是不同的问题约束导致了不同的最优解。**

## 约束驱动的架构选择

### Claude Code：本地优先，终端为王

**约束：**
- 运行在开发者的终端里
- 需要直接操作文件系统
- 流式 UX 要求高（Token 逐字显示）
- 权限安全要求高（可能执行危险命令）

**因此选择：**
- React + Ink 渲染终端（组件化 UI）
- 自建 Store + useSyncExternalStore（精确订阅避免闪烁）
- 四层权限模型（Deny > Allow > 工具自定义 > 用户交互）
- 本地优先的工具执行（BashTool、FileEditTool 等直接操作本地）

### DeepFlow：多步骤自动化，状态图驱动

**约束：**
- 需要执行多步骤的研究工作流
- 步骤之间有复杂的状态依赖
- 运行在服务端，不需要终端 UI
- 需要 Web API 供外部调用

**因此选择：**
- Python + LangGraph（状态图管理复杂工作流）
- 两行代码定义工具（简单直观，不需要终端渲染）
- Web API 接口

### NanoClaw：消息驱动，异步为王

**约束：**
- 通过消息渠道（WhatsApp/Telegram）交互
- 用户不会一直盯着看
- 任务可能很长（小时级）
- 没有 UI，纯消息

**因此选择：**
- Node.js + Claude Agent SDK
- 无 UI，SQLite 轮询触发对话
- 15 秒自动后台化（异步是默认）
- 消息级别的状态管理

## 三者的共同模式

尽管技术选择不同，三个系统共享几个核心模式：

1. **事件驱动循环** — 都是「接收输入 → 调用 LLM → 执行工具 → 循环」
2. **工具抽象层** — 都有工具注册、Schema 验证、结果回注
3. **上下文管理** — 都面临 Token 窗口限制，都需要压缩/记忆机制
4. **权限边界** — 都需要控制 LLM 能做什么、不能做什么

**理解这些共同模式，你就理解了 Agent Runtime 的本质。理解它们在不同约束下的不同选择，你就理解了工程权衡的艺术。**
