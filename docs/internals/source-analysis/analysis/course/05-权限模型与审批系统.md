# 第五集：权限模型与审批系统

同学们好，欢迎来到第五课——权限模型与审批系统。

这节课我们要搞清楚一件事：当 Claude Code 执行一个有风险的操作时，它到底听谁的？是你直接批准，还是管理员说了算，还是 AI 自己决定？

学完这节课，你会彻底理解 Claude Code 权限系统的设计哲学。无论你是日常使用还是做企业安全合规，都能心中有数。

## 学习目标

1. 搞懂四种权限模式各有什么用
2. 搞清楚工具被调用时，权限决策是怎么一步一步走下来的
3. 理解永久规则和会话记忆是怎么回事

## 场景思考

想象一个场景：你让 Claude Code「帮我清理一下项目里的临时文件」。Claude 想了想，决定执行 `rm -rf tmp/ build/`。

这时候你想要什么行为？

- **A**：直接执行，不问我
- **B**：弹出来问一句「确认吗？」
- **C**：先告诉我它打算干什么，我批准了才执行
- **D**：这个 `rm` 命令永远不能执行，太危险了

其实答案取决于你的工作场景、项目的重要程度、还有你对 Claude 的信任程度。而 Claude Code 的权限系统，就是为了支持所有这四种需求而设计的。

## 四种权限模式

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| **Default** | 有副作用的操作弹确认框 | 日常使用，控制力度最高 |
| **Auto** | 分类器自动判断是否批准 | CI/CD 环境，无人值守 |
| **Plan** | 必须先展示计划，批准后才执行 | 复杂任务，如重构认证系统 |
| **Bypass** | 不检查任何权限，直接执行 | 完全信任的环境 |

这四种模式可以理解为安全策略的四个档位：从最严格到最宽松，总有一款适合你当前的工作场景。

## 权限决策链

当一个工具真正被调用时，权限判断分**五个步骤**：

### 第一步：权限模式检查

- `Bypass` 模式 → 直接批准
- `Plan` 模式但不在 Plan 状态 → 直接拒绝

### 第二步：Deny Rules 检查（拒绝规则）

**这是绝对否决。** 一旦命中，任何 Allow Rule、任何用户操作都无法覆盖它。比如 MDM 下发了一条禁止 Bash 的策略，连 Bypass 都绕不过去。

### 第三步：Allow Rules 检查（允许规则）

匹配到允许规则就批准。

### 第四步：工具自定义逻辑

每个工具可以实现自己的 `checkPermissions` 方法做额外检查。比如 BashTool 可以检查命令是否在白名单里。

### 第五步：用户确认框

前面都没搞定，那就弹出确认框。等你点 Approve / Reject / Always Allow / Never Allow。

### 优先级口诀

```
Deny > Allow > 工具自定义 > 用户交互
```

## 权限规则的三个来源

| 层级 | 存储位置 | 作用范围 | 示例 |
|------|----------|----------|------|
| **Global（全局）** | `~/.claude/settings.json` | 所有项目 | 「永久允许 Read 操作」「永久禁止 Bash」 |
| **Project（项目级）** | `<project>/.claude/settings.json` | 单个项目，团队共享 | 「允许运行 `npm run test` 和 `npm run build`」 |
| **User（会话级）** | 内存 | 本次会话有效，结束消失 | 「这次允许删除 build 目录」 |

**优先级：Deny > Allow（无论哪个层级）。**

### 为什么要三层设计？

举个例子：项目规则允许 `npm run test`，但今天你在调试一个严重 Bug，不想让测试打断你。你可以在会话级加一个临时 Deny 覆盖项目级的 Allow。会话结束后，项目规则自动恢复。

这就是三层设计带来的灵活性。

## 永久规则的代价

当你点击「总是允许」时，Claude Code 做了两件事：

1. 把这条规则写入 `settings.json`，永久保存。
2. 同时更新内存中的 AppState，让新规则立即生效。

**「总是允许 Bash」是一个不可逆的操作。** 在你手动删掉这条规则之前，之后的 Claude 执行任何 Bash 命令都不会再问你——包括删除命令，包括被 Prompt Injection 注入的恶意命令。

这就是为什么权限系统里有分层设计：**Deny 是安全带，Allow 是油门。把油门踩到底之前，确保安全带系好了。**

## Prompt Injection 与权限安全

特别要注意：Claude Code 执行工具时不会区分「用户直接要求的命令」和「被 Prompt Injection 诱导的命令」。在它的视角里，所有命令都来自同一个对话上下文。

所以：
- **永远不要在 Bypass 模式下处理不可信的输入**
- **Deny Rules 是你防御 Prompt Injection 的最后一道防线**
- **MDM 下发的 Deny 规则连 Bypass 都绕不过去**——这是为企业安全设计的硬隔离
