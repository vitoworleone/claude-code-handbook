# 第6章 需求驱动工作流

Claude Code 的真正价值不在于"写代码更快"，而在于它能把一个模糊的想法变成可执行的、可验证的、可重复的工程流程。本章整合刘小排的需求驱动方法、Matt Pocock 的工程管线、徐文浩的实战案例，给出一条从"有个想法"到"代码上线"的完整路径。

---

## 6.1 需求明确阶段：五步消除歧义法

### 核心认知：Claude Code 是工作流执行引擎

不要把 Claude Code 当成"Cursor 升级版"或"自动补全工具"。它的本质是**工作流执行引擎**——你给足上下文和明确指令，它按流程执行；你给模糊需求，它只能猜，猜出来的就是屎山。

**关键洞察**：AI 的能力上限 = 上下文质量。你给它的信息越完整、歧义越少，输出质量越高。

### 五步流程详解

**第一步：飞书写完整需求文档**

不要直接打开 Claude Code 说"帮我做个登录功能"。先花 15-30 分钟写一份完整的需求文档：

- 用户场景（谁在什么情况下用这个功能）
- 功能范围（做什么）
- 明确写出 Out of Scope（不做什么）
- 验收标准（怎么算完成）
- 原型图或流程图（哪怕手绘拍照）

文档长度一两千字为宜。太短歧义多，太长没人看。

**第二步：让 Claude 复述理解**

把需求文档贴给 Claude，让它复述对你的需求的理解。这一步不是为了测试 Claude，而是为了**暴露歧义**。

```
> 请复述你对这个需求的理解，包括：
> 1. 这个功能解决什么问题
> 2. 涉及哪些模块
> 3. 关键决策点有哪些
```

如果复述和你的意图有偏差，说明文档有歧义。修正文档，而不是在实现过程中纠正。

**第三步：让它出技术方案，你审查**

```
> 基于以上需求，给出技术实现方案。包括：
> 1. 涉及哪些文件的修改
> 2. 数据流怎么变化
> 3. 潜在风险点
```

**关键点**：技术方案的决策权在你。Claude 可以提议，但风险高的决策（如引入新依赖、修改数据库 schema）必须由人拍板。

**第四步：让它反向提问，你拍板决策**

```
> 作为实现者，你有哪些问题需要我决策？
> 请列出所有有风险、有歧义、需要取舍的点。
```

这一步是防屎山的核心。让 Claude 把所有"它只能猜"的点暴露出来，你逐个拍板。比如：
- "用户密码用 bcrypt 还是 argon2？"
- "失败重试几次？间隔多久？"
- "老数据怎么迁移？"

**第五步：讨论到毫无疑义后再写代码**

只有当以上所有问题都有明确答案，才进入实现阶段。如果在实现过程中发现还有歧义，停下来回到第一步补充文档，而不是边做边改。

**底层原理**：表达有歧义 → 模型只能猜 → 屎山。让有风险、需决策的点全部由人来决策，Claude 只负责执行已确定的方案。

## 6.2 PRD 与看板：从模糊想法到可执行任务

### PRD 是"旅程文档"，不是规格书

Matt Pocock 强调：PRD（Product Requirements Document）不是写一份完美无缺的规格书然后丢给 AI 去实现。它的作用是**记录目的地和关键决策**，作为 AI 执行时的参考。

**PRD 应该包含**：
- 背景和目标（为什么做）
- 用户流程（怎么做）
- 关键决策（已确定的方案选择）
- Out of Scope（明确不做的范围）
- 验收标准（怎么算完成）

**PRD 不应该包含**：
- 详细的 API 设计（AI 可以自己做）
- 具体的代码实现（那是实现阶段的事）
- 完美的文学性描述（浪费时间）

> **花时间的地方是 QA，不是 PRD。** PRD 写得差不多就可以开始，细节在实现和 QA 中迭代。

### Out of Scope 极其重要

这是最容易被忽视但最关键的部分。明确告诉 Claude"这个不做"，比告诉它"做这个"更重要。因为 AI 的默认行为是"多做一点以防万一"，如果没有明确边界，它会实现很多你不需要的功能。

**示例**：

```markdown
## Out of Scope
- 不处理第三方 OAuth 登录（仅支持邮箱+密码）
- 不实现密码找回（后续迭代）
- 不做 rate limiting（基础设施层已处理）
```

### 看板 = DAG（有向无环图）

把 PRD 拆解成看板（GitHub Issues、Linear、Jira 等）时，核心原则是**垂直切片**和**DAG 结构**。

**垂直切片 vs 水平分层**：

| 水平分层（❌） | 垂直切片（✅） |
|--------------|--------------|
| Issue 1: 设计数据库表 | Issue 1: 用户注册（含 DB + API + UI） |
| Issue 2: 写 API 接口 | Issue 2: 用户登录（含 DB + API + UI） |
| Issue 3: 做前端页面 | Issue 3: 权限检查（含 DB + API + UI） |

水平分层的问题是：AI 做完 Issue 1 后，Issue 2 和 3 可能做不下去（依赖没对齐）。垂直切片让每个 issue 都是可独立交付的完整功能。

**blocked_by：DAG 的核心机制**

每个 issue 明确写出它依赖哪些 issue：

```yaml
# Issue: 用户登录
blocked_by:
  - "用户注册"  # 必须先有注册，才能登录
```

Claude Code 可以读取看板的 blocked_by 关系，自动按 DAG 顺序执行。没有 blocked_by 的 issue 可以并行处理。

**什么时候进 AFK（黑夜班）**

不是每个 issue 都适合自动执行。只有满足以下条件的才进 AFK：
- 边界清楚（In Scope 和 Out of Scope 明确）
- 验收标准可验证（有明确的"完成"定义）
- 风险可控（不涉及数据库迁移、安全敏感操作等）

边界模糊、需要人机协作的 issue 标记为 HITL（Human In The Loop），由人主导。

## 6.3 Plan Mode：复杂任务先规划再执行

### 什么是 Plan Mode

Plan Mode 是 Claude Code 的一种权限模式，告诉 Claude"先调研、再规划、最后才执行"。在这个模式下，Claude 可以读取文件、运行命令来理解问题，但**不会修改任何文件**。

**进入方式**：
- 启动时：`claude --permission-mode plan`
- 会话中：按 `Shift+Tab` 切换到 plan 模式
- 单条提示：`/plan 你的需求`

### 完整操作流程

**阶段一：调研与规划**

Claude 读取相关文件、运行探索命令，然后输出一份详细的实施计划。计划通常包括：
- 需要修改哪些文件
- 每个文件的修改内容
- 潜在的依赖关系和冲突
- 建议的验证步骤

**阶段二：审查与编辑**

你可以直接审查这份计划，也可以用 `Ctrl+G` 在默认编辑器中打开并修改：

```
[Claude 输出计划...]

> 按 Ctrl+G 在编辑器中打开计划，修改后保存退出
```

在编辑器中，你可以：
- 删除不需要的修改
- 调整实现顺序
- 补充遗漏的边界情况
- 修改验收标准

**阶段三：批准与执行**

审查完成后，Claude 提供批准选项：

| 选项 | 效果 |
|------|------|
| 批准并进入 auto 模式 | 后续所有操作自动执行（分类器安全检查） |
| 批准并进入 acceptEdits 模式 | 自动批准文件编辑，其他操作仍需确认 |
| 批准并手动审查每个编辑 | 每个文件修改前都询问 |
| 继续规划 | 给反馈，让 Claude 改进计划 |
| 精炼（Ultraplan） | 在浏览器中打开计划，可视化审查 |

批准后，Claude 退出 plan 模式，切换到选择的执行模式，开始实施计划。同时，会话会自动命名（基于 plan 内容），除非你已手动命名。

### Plan Mode 与 PRD 的关系

| | PRD | Plan Mode |
|--|-----|-----------|
| 谁写 | 人 | Claude（基于人的需求） |
| 粒度 | 功能级 | 文件级 |
| 作用 | 对齐人和 AI 的理解 | 把 PRD 翻译成可执行步骤 |
| 时机 | 需求阶段 | 实现前 |

**典型工作流**：
1. 人写 PRD（功能级规划）
2. Claude 进入 Plan Mode，读取 PRD 和代码，输出文件级计划
3. 人审查并编辑计划
4. 批准执行

### 什么时候用 Plan Mode

**必须用**：
- 探索新代码库（先理解架构再动手）
- 大规模重构（不确定影响范围）
- 跨多个模块的复杂功能（需要协调多个文件）

**可以跳过**：
- 范围明确的小修复（拼写错误、重命名、加日志）
- 单一文件的修改
- 重复性任务（已有明确模式）

## 6.4 实现阶段：AFK 代理、深模块与规范策略

### AFK 代理（黑夜班 / Ralph Loop）

AFK（Away From Keyboard）代理是 Matt Pocock 工作流的核心——把验证过的流程变成后台自动执行的任务。

**核心脚本思路**：

```bash
#!/bin/bash
# once.sh - 单次执行脚本

# 读取当前可执行的 issues
ISSUES=$(cat issues/*.md)

# 读取最近 commit 作为上下文
LAST_COMMITS=$(git log --oneline -5)

# 加载 implementer prompt
PROMPT=$(cat prompts/implementer.md)

# 调用 Claude Code
echo "$ISSUES" | claude \
  --permission-mode acceptEdits \
  --prompt "$PROMPT" \
  --context "Last 5 commits: $LAST_COMMITS"
```

**implementer prompt 的核心要素**：

```markdown
# Implementer Prompt

## 任务
读取看板中标记为 AFK 的 issues，按优先级实现。

## 优先级（从高到低）
1. Critical bugfixes - 影响用户的严重 bug
2. Dev infra - 开发环境、工具链
3. Tracer bullets - 垂直切片的端到端功能
4. Refactors - 代码重构

## 规则
- 只处理 AFK issues，跳过 HITL
- 每个 issue 完成后运行测试验证
- 使用 `<promise>NO MORE TASKS</promise>` 标记完成
- 如果测试失败，修复后再继续
```

**四级优先级详解**：

| 优先级 | 内容 | 为什么优先 |
|--------|------|-----------|
| Critical bugfixes | 影响用户的严重 bug | 线上问题，必须立即修 |
| Dev infra | 开发环境、工具链 | 阻塞其他工作的基础设施 |
| Tracer bullets | 垂直切片的端到端功能 | 快速验证方向正确 |
| Refactors | 代码重构 | 改善代码质量，不阻塞功能 |

**`<promise>NO MORE TASKS</promise>`**：这是一个结构化状态信号。当 Claude 输出这个标记时，表示当前批次所有 AFK issues 已处理完毕。你的外层脚本可以检测这个标记，决定是否继续轮询新 issues。

### 深模块架构：为 AI 优化代码结构

AI 处理代码的方式和人不同。人可以通过记忆和直觉在大量小文件间跳转，AI 的上下文有限，必须在给定上下文中理解代码。

**浅模块（AI 地狱）vs 深模块（AI 天堂）**：

| | 浅模块 | 深模块 |
|--|--------|--------|
| 文件数量 | 大量小文件 | 少量大文件 |
| 依赖结构 | 网状依赖 | 树状层次 |
| 导出数量 | 每个文件 1-2 个 | 每个文件 5-10 个 |
| 测试方式 | 需要大量 mock | 围绕模块边界测试 |
| AI 导航难度 | 极高（上下文碎片化） | 低（清晰的入口和出口） |

**灰盒模式**：

这是 AI 时代的新型开发模式：

1. **你设计模块接口**（输入、输出、行为契约）
2. **把实现委托给 AI**
3. **你只需要知道"做什么"，不需要知道"怎么做"**

**关键警告**：AI 默认产生浅模块（大量小文件、每个文件只做一件事）。你必须在 PRD 阶段就明确模块结构，要求 AI 按深模块方式组织代码。

**示例**：

```markdown
## 模块设计要求

### auth 模块（深模块）
- 入口：`auth/login`, `auth/register`, `auth/logout`
- 内部包含：验证逻辑、token 生成、session 管理
- 不暴露内部函数
- 所有外部调用通过三个入口

### 禁止
- 把验证逻辑拆成 `validators/email.ts`, `validators/password.ts`
- 把 token 逻辑拆成 `tokens/jwt.ts`, `tokens/refresh.ts`
```

### Push vs Pull 规范策略

在多 Agent 协作时，规范（编码标准、架构约束）的传递方式影响效率和准确性。

| | 实现者 (Implementer) | 审查者 (Reviewer) |
|--|---------------------|------------------|
| 模型 | Sonnet（快） | Opus（聪明） |
| 规范传递 | **PULL**（按需拉取 skill） | **PUSH**（强制注入全部规范） |
| Prompt 长度 | 短（只含任务描述） | 长（含全部规范和上下文） |
| 为什么 | 实现时不需要知道所有规则，只需要当前任务相关的 | 审查必须对照全部规范检查 |

**审查比实现更需要"聪明"**，因为审查只有一次机会发现问题。如果审查 agent 漏掉了规范违规，代码就会合入主干。

### 实战案例：网页端到端测试生成（徐文浩）

**场景**：需要为 Web 应用生成端到端测试，但传统的录制回放工具维护成本高。

**两阶段法**：

**阶段一：探索**

让 AI 在浏览器中自己操作应用，记录操作步骤：

```
> 打开我们的登录页面，尝试正常登录流程。
> 记录你的每一步操作（点击、输入、跳转）。
> 不要写测试代码，只记录操作日志。
```

AI 输出：

```markdown
## 探索记录

1. 访问 /login
2. 输入 email: test@example.com
3. 输入 password: password123
4. 点击 "Login" 按钮
5. 页面跳转到 /dashboard
6. 检查页面包含 "Welcome, test"
```

**阶段二：转化为测试**

```
> 基于以上探索记录，生成 Playwright 测试脚本。
> 每个操作对应一个测试步骤。
> 添加断言验证关键元素存在。
```

**核心洞察**：把 AI 当成人，按人的方式干活最靠谱——先探索理解，再总结执行。

## 6.5 审查与 QA：品位把关与互掐闭环

### QA 不是测试，是品位把关

传统 QA 是测试工程师写用例、跑回归。在 AI 工作流中，QA 的含义变了：

| | 传统 QA | Matt 的 QA |
|--|---------|-----------|
| 执行者 | 测试工程师 | **你自己** |
| 方法 | 写用例、跑回归 | **亲自上手用、检查体验** |
| 时机 | 开发完成后 | **每次代理产出后立刻** |
| 输出 | Bug 报告 | **看板中的新 issue** |

**关键原则**：QA 中发现的问题**永远不要自己去改**。把问题描述成新 issue，丢回看板，让 AFK 代理去修。为什么？因为：
- 人修一个 bug 花 5 分钟，但打断了当前工作流
- 把 bug 变成 issue，代理在后台修，人不中断
- 强制你把问题描述清楚（写 issue 的过程就是梳理思路）

### AI 写 + AI 审互掐闭环

这是徐文浩提出的核心实践：让两个 AI agent 互相审查。

**为什么不能用同一个 AI 审查自己？**

同一个 Claude Code 会话写完代码后，上下文已经被"消耗"了——它记得自己的实现思路，带着偏见审查。这被称为 **Dumb Zone**（上下文已消耗区）。

**正确做法**：

1. **写代码 agent**：在会话 A 中实现功能
2. **审查 agent**：在会话 B（全新上下文）中审查代码
3. **两个 agent 完全独立**，审查 agent 不知道实现 agent 的思路

**自动化互掐流程**：

```
实现 Agent 完成代码 → 提交 PR → 触发审查 Agent
                                    ↓
                              审查 Agent 读取 diff + PLAN.md
                                    ↓
                              输出 approve / reject + 评论
                                    ↓
                              reject → 通知实现 Agent 修复
                              approve → 进入合并流程
```

**审查 prompt 要点**：

```markdown
审查要求：
1. 检查每个需求是否实现
2. 检查边界情况是否处理
3. 检查是否有 out-of-scope 的修改
4. 报告缺陷，不报告风格偏好
```

### 发现 Bug 的处理策略

当 AI 在补自动化测试时发现 bug，有两种处理策略：

**策略 A（保守）**：注释掉失败的测试，标记 `TODO`，另开分支修复。
- 优点：每个提交只解决一个问题，commit 历史干净
- 缺点：bug 暂时存在，需要后续跟进

**策略 B（激进）**：顺手修掉。
- 优点：问题立即解决
- 缺点：一个提交混杂了多个改动，回滚困难

**徐文浩的建议：选 A**。AI 的第一时间想法可能与人不一致，每个提交只解决一个问题，保持代码历史的清晰性。

## 6.6 完整管线：从想法到上线

### 顺序管线

```
想法 → 烘培对话 → PRD → 看板 → AFK代理 → QA → 代码审查 → 上线
  ↑                                    ↓
  └────────── 改进代码架构 ← 架构审查 ←─┘
```

**各阶段详解**：

| 阶段 | 执行者 | 输出 | 关键动作 |
|------|--------|------|---------|
| 想法 | 人 | 模糊的需求 | 头脑风暴、用户反馈 |
| 烘培对话 | 人 + AI | 对齐的理解 | AI 连环提问，人回答决策 |
| PRD | 人 | 需求文档 | 写清楚做什么、不做什么 |
| 看板 | 人 | DAG 结构的 issues | 垂直切片、写 blocked_by |
| AFK 代理 | AI | 代码提交 | 按优先级自动实现 |
| QA | 人 | 新 issues 或 approve | 亲自上手用、检查体验 |
| 代码审查 | AI | approve / reject | 独立 context 审查 |
| 上线 | 人 + AI | 生产环境 | 合并、部署、验证 |

**烘培对话（Grilling Session）**：

在 PRD 之前，花 10-20 分钟和 AI 进行"烘培式对话"。目的不是产出文档，而是**建立 shared understanding**：

```
> 我想做一个用户登录功能。在写 PRD 之前，我需要确保我们理解一致。
> 请你作为实现者，问我所有需要明确的问题。
> 每个问题都要具体到可以影响实现决策。
```

AI 会连环提问：
- "用户是用邮箱+密码还是支持 OAuth？"
- "密码最小长度多少？"
- "登录失败几次后锁定账户？"
- "session 过期时间多久？"

你的回答直接成为 PRD 中的关键决策。

### 并行化流水线（四角色模型）

当项目规模扩大，顺序执行不够快，可以引入并行化流水线：

```
Planner → 读取看板 DAG → 输出"下一批可并行的 issues"
    ↓
Sandbox #1 / #2 / #3（git worktree 隔离）
  → Implementer (Sonnet, PULL skills)
  → Reviewer (Opus, PUSH 规范)
    ↓
Merger → 合并所有分支 → 解决冲突 → 跑全量测试
```

**四个角色**：

| 角色 | 职责 | 工具/模型 |
|------|------|----------|
| Planner | 读取看板，找出可并行执行的 issues 批次 | Claude Code |
| Implementer | 在隔离 worktree 中实现代码 | Sonnet + PULL skills |
| Reviewer | 审查代码，对照规范检查 | Opus + PUSH 规范 |
| Merger | 合并分支、解决冲突、跑测试 | Claude Code + Bash |

**Sandbox 隔离**：

每个 Implementer 和 Reviewer 运行在独立的 git worktree 中：

```bash
# Planner 分配任务
claude --worktree sandbox-1 --bg "implement issue #123"
claude --worktree sandbox-2 --bg "implement issue #124"
claude --worktree sandbox-3 --bg "implement issue #125"
```

工作完成后，Planner 检查状态，Merger 负责合并到主分支。

**演进路线**：

| 阶段 | 特征 | 适合场景 |
|------|------|---------|
| 手动 | 人逐个实现、审查、合并 | 小项目、探索期 |
| 顺序 Ralph Loop | 一个 AFK 代理顺序执行 | 中等项目、稳定流程 |
| 带审查的 Ralph Loop | AFK 实现 + AI 审查 | 较大项目、需要质量保证 |
| 并行工厂 | 多 sandbox 并行 | 大项目、高并发需求 |

### 实战案例：飞书 Bug 自动收集（徐文浩）

**场景**：团队使用飞书接收 bug 报告，但报告散乱、重复、描述不清，人工整理耗时。

**AI 自动做四件事**：

1. **收集**：扫描飞书群所有 bug 报告，读取消息内容
2. **去重**：合并同义描述的 bug（"登录不了"和"无法登录"识别为同一个）
3. **追问**：描述不清的自动找提报人问清楚
   ```
   "您好，您报告的'页面报错'需要更多信息：
   1. 哪个页面？URL 是什么？
   2. 报错的具体内容或截图？
   3. 复现步骤？"
   ```
4. **输出**：整理后写入项目管理工具（Jira、Linear、GitHub Issues）

**前提**：各系统权限配好（飞书机器人、项目管理工具 API）。

**核心洞察**：遇到 AI 干不了的事，思考"应该给它补什么权限"。不是"AI 不能做这个"，而是"我还没给 AI 配齐工具"。

## 6.7 规模化：校准→批量→全自动

### 三阶段演进

| 阶段 | 模式 | 人力投入 | AI 自主度 |
|------|------|---------|----------|
| 校准期 | 人工 + AI 并行 | 高 | 低 |
| 过渡期 | AI 做，你抽查 | 中 | 中 |
| 放手期 | 全自动 | 低 | 高 |

**校准期**：

你做一个，AI 做一个，对比结果，纠正 AI 的错误。这个阶段通常需要 **30-60 个模板**的验证。

示例：
- 你："帮我写一个 React 组件，显示用户列表"
- Claude 生成代码
- 你审查："这个应该用 useMemo 优化，不要直接 map"
- 把纠正写进 CLAUDE.md
- 下一个类似任务，检查 Claude 是否记住了

**过渡期**：

AI 做，你抽查。不逐个审查，随机抽样检查质量。

**放手期**：

AI 自己跑，零人力。你只在异常时介入。

> **前面 30-60 个模板验证通过后，后面放心全自动。**

### 核心思维转变

**不要想**："这一轮产出什么代码？"

**要想**："这个流程能不能跑 100 次？能不能批量验证？验证通过能不能后台自己跑？"

**人力花在"打磨流程"，不是"执行流程"**。花 2 小时优化一个 prompt，让它在接下来 100 次执行中都正确，比花 2 小时执行 10 次任务更有价值。

### 子代理与 Sandbox

刘小排推荐的第二个核心功能是**子代理 / Sandbox**。

**给每个 Agent 分配专精领域**：

```
Agent: image-style-generator
- 专精：图片风格词生成
- Prompt："你是一个图片风格专家，只生成 Midjourney 风格的描述词"
- 工具：仅需要 Bash 和 Read
- 运行方式：后台常驻
```

**独立上下文窗口**：每个子代理有自己独立的上下文，不受主会话影响。

**特定工具访问**：可以给子代理限制工具权限，比如只允许 Read 和 Bash，不允许 Edit。

**使用场景**：
- 把探索任务隔离到子代理（不污染主上下文）
- 给不同领域分配不同专家
- 后台常驻服务（如 bug 收集、监控报警处理）

### 输出风格

刘小排推荐的第三个核心功能是**输出风格（Output Styles）**。

Claude Code 支持四种输出风格：

| 风格 | 特点 | 适用场景 |
|------|------|---------|
| Default | 标准回答 | 日常开发 |
| Proactive | 主动建议下一步 | 探索性任务 |
| Explanatory | 详细解释每一步 | 学习新代码库 |
| Learning | 留 TODO(human) 让你自己填 | 教学、培养能力 |

**Learning 模式**特别值得展开：

在 Learning 模式下，AI 不会把代码写完整，而是：

```typescript
// AI 生成的代码
function calculateTotal(items: CartItem[]) {
  // TODO(human): 实现计算逻辑
  // 提示：考虑折扣、税费、运费
  throw new Error("Not implemented");
}
```

这种方式强迫你理解每一行代码，而不是被动接受。适合：
- 学习新技术
- 代码审查培训
- 培养 junior developer

## 6.8 模型分工策略

不要用一个模型做所有事。不同模型有不同优势，合理分工能大幅提升效率。

### 三层分工模型

```
Gemini（1M 上下文）→ 架构设计、方案决策
        ↓
Claude Code → 执行 90% 日常开发
        ↓
GPT-5/O3/O4 Mini → 高难度算法、数学/科研代码
```

| 模型 | 优势 | 职责 |
|------|------|------|
| **Gemini** | 1M 上下文窗口 | 架构设计、读取大量代码库、方案决策 |
| **Claude Code** | Agent Runtime、工具集成 | 日常编码、文件操作、测试运行、Git 操作 |
| **GPT-5/O3** | 推理能力最强 | 算法攻坚、数学推导、科研代码、复杂逻辑 |

**实际分工示例**：

**场景**：实现一个分布式锁

1. **Gemini**："读取我们的整个代码库，分析现有并发控制方案，建议分布式锁的集成点"
   - 利用 1M 上下文读取大量文件
   - 输出架构建议

2. **Claude Code**："基于以上架构，实现 RedisDistributedLock 类，包括加锁、解锁、超时处理"
   - 编辑文件
   - 运行测试
   - 提交代码

3. **GPT-5**（如果锁算法有复杂数学）："验证这个锁算法的正确性，检查是否有竞态条件"
   - 深度推理
   - 发现边界情况

**何时切换模型**：

| 信号 | 切换到 |
|------|--------|
| Claude Code 反复搞不定一个算法 | GPT-5/O3 |
| 需要理解整个大型代码库的架构 | Gemini |
| 日常编码、文件操作、测试 | Claude Code |
| 代码审查发现复杂逻辑问题 | Opus（Claude 系列中最强） |
