# 第8章 开发节奏与心态

前两章讲了工作流和安全架构——这些都是"术"。本章讲"道"：使用 Claude Code 带来的根本性角色转变，以及如何调整心态适应这种转变。

---

## 8.1 三阶段消耗策略

第 6 章已经详细介绍了校准→过渡→放手的三阶段模型。本章不再重复操作细节，而是讲每个阶段的心理调整。

### 校准期：学会"容忍浪费"

**典型心态**：

"我做这个 5 分钟就搞定了，让 AI 做却要花 20 分钟纠正它，太慢了。"

**正确心态**：

校准期的"慢"是投资。你现在花 20 分钟训练 AI，是为了之后 100 次执行每次省 4 分钟——净赚 380 分钟。

**操作要点**：
- 前 30-60 个模板必须人工验证，不要跳过
- 每次纠正都写进 CLAUDE.md，确保 AI 下次不再犯
- 校准期的质量决定放手期的可靠性

一个具体数字：刘小排的 Fast 3D 项目中，校准期花了约 30 个模板的验证，之后数百个模板全自动生成了满意的结果。

### 过渡期：学会"抽查而非全查"

**典型心态**：

"AI 做的我不放心，每个都要看一遍。"

**正确心态**：

如果有 30 个验证通过的模板在前，抽查就够。你的目标是找到"AI 容易出错的模式"，而不是找到"每一个错误"。

**抽查方法**：
- 随机抽取 10-20% 的产出检查
- 重点关注新类型的任务（AI 没见过的模式）
- 如果连续 10 个抽查都无大问题，提升抽查间隔

### 放手期：学会"只介入异常"

**典型心态**：

"我还是不太放心，要不我再看看？"

**正确心态**：

放手期的你就像自动驾驶的监督者——正常行驶时你不需要碰方向盘。只在报警响起时才介入。

**介入标准**：
- 测试失败
- 代码审查 reject
- 用户反馈 bug
- 你不理解的决策（分级关注中风险大的部分）

## 8.2 人力花在"打磨流程"，不是"执行流程"

这是刘小排和 Matt Pocock 共同强调的核心思维转变。

### 打磨流程 vs 执行流程

| | 执行流程 | 打磨流程 |
|--|---------|---------|
| 做什么 | 亲自写代码、改 bug | 优化 prompt、完善 CLAUDE.md |
| 产出 | 一个功能 | 一个能批量产出功能的系统 |
| 可复制性 | 低（只有你懂） | 高（团队其他人也可以用） |
| 边际成本 | 不变（每次都要你） | 递减（越用越好） |
| 时间尺度 | 分钟/小时 | 月 |

### 具体例子

**场景**：项目有 50 个 Component 需要从 class component 迁移到 hooks。

**执行流程思维**：
- 自己一个一个改，一天改 10 个，5 天完成
- 每个都亲自动手，质量有保障
- 但下次再有 50 个，又得手动

**打磨流程思维**：
- 先花半天写好 prompt 和 CLAUDE.md 规范
- 用 5 个 Component 做校准，验证 prompt 效果
- 把剩下的 45 个丢给 AFK 代理，后台跑
- 期间你去做更有价值的事
- 下次再有类似迁移，直接复用这套 prompt

**结果**：打磨流程的一次性投入（半天），换来了数百次的免人工执行。

### 判断标准：这个工作你会做第二次吗？

每次做重复性工作时，问自己：
- 这个流程会不会出现第二次？
- 如果会，有没有办法让 AI 记住这个流程？
- 如果 AI 记住了，下次我能不能只做抽查？

> 答案通常是：写进 CLAUDE.md，做成 skill，或做成可复用的 prompt 模板。

### 一个容易被忽略的原则：Harness 也要定期简化

本章一直在讲"打磨流程"——优化 prompt、完善 CLAUDE.md、设计工作流。但有一个对称的原则同等重要：**流程也需要被删除。**

Anthropic 工程团队的 harness 设计文章中记录了一个具体案例：

> 从 Opus 4.5 升级到 Opus 4.6 后，他们移除了 sprint construct 和 context resets——成本从 **$200 降到了 $125**，同时输出质量保持不变。

核心洞察是：**"Every harness component encodes an assumption about what the model cannot do on its own, and those assumptions go stale as models improve."**

翻译：**每一个流程组件都在编码一个"模型自己做不了什么"的假设，而随着模型改进，这些假设会过时。**

这意味着："打磨流程"不是一个只增不减的过程。每次模型升级（`/model opus` 或等待 Anthropic 更新 Sonnet/Opus 底层模型），你都应该审视：

- 哪些检查步骤是以前模型弱时需要、现在模型够聪明了可以省略的？
- 哪些 context reset 是以前的低质量长对话需要、现在可以去掉以节省 token 的？
- 哪些 prompt 里的详细指令是以前模型不理解、现在模型已经内化了的？

不删旧的流程组件，你不仅浪费 token，更浪费你的注意力——你在审查 AI 已经能完美处理的事情。

## 8.3 "你是主语，AI 是动词"——协作心态的外部视角

第 8.2 节讲的"打磨流程而非执行流程"，以及第 6 章刘小排的流水线方法，都是软件工程领域的内部经验。本节引入一个**跨领域的外部视角**——来自内容创作者汉洋和重轻（播客《诗梳风》）在大量非虚构写作中与 AI 协作的实战经验。他们的工作流与软件工程差异很大（写 18000 字长文 vs. 写代码），但底层心态原则高度相通。这些原则可以验证你对 8.2 的理解，同时补充一些在工程视角下容易被忽略的认知盲区。

汉洋团队的核心结论可以概括为一句话：**"主语还是你，宾语还是对象，AI 只是动词里面的内涵。"** 以下是他们经过一年半实战提炼出的五条最关键原则，每条都附带"为什么"和"在 Claude Code 中怎么用"。

---

### 原则一：AI 是工作台，不是许愿池

**核心观点**：永远不要希望 AI 直接给你结果。你要把大模型理解成一个有一堆工具的工作台，而不是一个能直接产出成品的自动贩卖机。

**为什么这条重要**：

人们使用 Claude Code 时最常见的失败模式是"许愿式提问"——直接丢给它一句话："帮我重构这个模块"、"给我写个测试"。这种问法的问题在于：AI 不知道你的背景知识、不知道你的质量标准、不知道这件事在你的项目中的优先级。它只能猜，猜出来的就是第 6 章说的"屎山"。

汉洋团队用一个极端例子说明了这个问题：对着 AI 说"请给我解释这篇研究"就是一个愿望，而非任务。因为"解释"对不同人有完全不同的含义——给教授看的解释、给博士生看的解释、给外行人看的解释，是三种完全不同的输出。没有明确约束的"解释"请求，AI 只能输出一种平庸的、对谁都不合适的中间态。

**在 Claude Code 中的实践**：

把每一个任务都拆解成"背景 + 目标 + 约束 + 输出格式"四要素：

| 要素 | 许愿式提问 | 工作台式提问 |
|------|-----------|-------------|
| 背景 | "帮我改这个文件" | "这个文件是用户认证模块，当前用 bcrypt，我们准备迁移到 argon2" |
| 目标 | "改好" | "把 password_hash 调用替换为 argon2id，保持现有接口不变" |
| 约束 | （无） | "不要修改测试文件，我会单独处理" |
| 输出格式 | （无） | "先给出变更 diff 预览，我确认后再执行" |

**关键洞察**：在 Claude Code 中，这个四要素不是一次性说给 AI 听的——你应该把它写进 CLAUDE.md 或做成 Skill（见第 12 章），让 AI 每次启动时自动加载。这样你就不需要每轮对话都重复一遍背景。

---

### 原则二：像尊重人一样尊重 AI

**核心观点**：把 AI 当成一个你付工资的秘书——它对你非常客气、尽心尽力，但有些东西它不懂，你需要从头到尾教它。

**为什么这条重要**：

很多人对 AI 同时持有两种矛盾态度：一方面觉得"AI 太像人了"，另一方面在让它干活时又把它当成全能神，期望它能读心。汉洋的观察是：如果你对着一个真人秘书，你不会只说"去帮我把我写俄罗斯种大豆那篇文章改好"——你会解释文章是写给谁看的、你现在卡在哪个部分、你想保留什么风格。对 AI 也应该如此。

这个原则的深层含义是：**你要反思自己，而不是否定 AI。** 当 AI 的输出不满意时，第一反应不应该是"AI 不行"，而应该是"我有没有把需求说清楚"。这是 8.1 校准期心态的另一种表述——校准期校准的不是 AI，是**你自己表达需求的能力**。

**在 Claude Code 中的实践**：

- 当 Claude 的代码不符合你的风格时，不要直接说"写得太烂了，重写"——打开 CLAUDE.md，把你的代码风格规范写进去，让它下次自动遵循
- 当 Claude 反复犯同一个错误时，不要每次对话中都口头纠正——把这个错误模式和正确做法写成一条 rule（见第 5 章 5.4 节）
- 当 Claude 做了你不理解的决策时，问它"你为什么这样设计？"，而不是直接否决。这个过程是你在理解它的"思考方式"，以便以后更好地指导它

---

### 原则三：改产线，不改结果

**核心观点**：克制住自己上手修改 AI 输出的冲动。你要改的是产线（prompt、CLAUDE.md、Skill），而不是单篇产出。

**为什么这条重要**：

这是汉洋团队的核心发现，与第 8.2 节"打磨流程 vs 执行流程"完全呼应。他们最初用 AI 写文章时，每篇产出都是 59 分，汉洋忍不住妙手一改就到 80 分——改到最后发现"还不如我自己写"，于是放弃了 AI。后来他们意识到：**问题不是 AI 不行，是自己一直在做错误的事。**

正确的做法是：看到 AI 产出 59 分时，忍住改它的冲动，分析"是哪个步骤导致只有 59 分"——是 prompt 太模糊？是缺少风格示例？是上下文给得不够？然后修改那个步骤的参数，重新跑一遍产线。目标不是让单次产出达到 100 分，而是让产线的**平均分**从 59 提升到 75-80。一旦平均分到了 75，批量生产时自然会有 85 甚至 90 分的惊喜。

**在 Claude Code 中的实践**：

与第 8.2 的具体例子形成对照：

| 场景 | 改结果（错误） | 改产线（正确） |
|------|--------------|---------------|
| Claude 写的函数命名不符合团队规范 | 手动把 `getData` 改成 `fetchUserProfile` | 在 CLAUDE.md 的"代码风格"章节写明命名规范，附 3-5 个示例 |
| Claude 生成的测试覆盖不全 | 手动补写缺失的边界测试 | 在 prompt 中增加"测试覆盖检查清单"，让 Claude 自检 |
| Claude 连续 3 次理解错同一个需求 | 放弃 AI，自己写 | 分析需求描述中哪句话有歧义，重写 PRD 中该段落 |

**关键数字**：汉洋团队的经验是，产线的每次改进会放大到后续 100 次执行。在 Claude Code 中，一个写好的 CLAUDE.md 规则会在每次对话中自动生效——这意味着你花 10 分钟打磨一条规则，可能节省未来 100 次对话中的 100 次纠正。

---

### 原则四：数量是第一要务

**核心观点**：不要省着用 AI。你要的不是一个 100 分的结果，而是一个稳定产出 70 分结果的机器——然后让它跑 100 次，从中挑最好的。

**为什么这条重要**：

人们使用 Claude Code 时的另一个常见错误是"小心翼翼"——每个月付 20 美元订阅费，结果每次只敢让 Claude 做一件事，做完觉得"还行"就停了。汉洋团队的做法完全相反：让他们报 50 个选题（每 10 个一组），让他们生成 5 个版本的开头，让他们写 10 个不同的测试方案——**大量产出，然后筛选。**

这个原则的数学基础是：如果 AI 的平均产出质量是 70 分，那么你让它生成 10 个版本，至少有一个会达到 80 分以上。而如果你只让它生成 1 个版本，然后自己花同样时间把它从 70 分改到 80 分——你不仅浪费了修改时间，还失去了"发现意外惊喜"的机会（AI 有时会在第 7 个版本里给出一个你完全没想到的好方案）。

**在 Claude Code 中的实践**：

- **方案生成**：不要问"怎么实现这个功能？"，而是问"给我 3 个不同的实现方案，分别说明优缺点"——然后你选最适合的
- **代码审查**：让 Claude 以 Reviewer agent 的身份给出审查意见，同时以另一个 Agent 的身份给出反驳意见——两个视角碰撞后，你做最终判断
- **Bug 排查**：遇到诡异 bug 时，不要只问一次"怎么修"。让 Claude 先给出 3 个可能的原因假设，然后逐一验证。这比单一路径的排查更快
- **AFK 批量任务**（见第 6 章 6.7 节）：这是"数量是第一要务"的工程化体现——把 50 个模板化的任务丢给后台代理，跑完后人工抽查 10%

---

### 原则五：不要越俎代庖——你是总厨，AI 拍黄瓜

**核心观点**：尊重 AI 的主体性。你的目标是让 AI 的思维模式稳定产出你要的东西，而不是替它完成每一个步骤。

**为什么这条重要**：

汉洋用了一个餐厅隐喻：你是行政总厨，你不会亲自去拍黄瓜。你尝一口，说"咸了，回去重做"，并且告诉厨师怎么调整——但你不会自己拿起刀来拍。很多人用 AI 时的挫败感来源于**模糊了管理者和执行者的边界**——看到 AI 做得不好，就直接上手改；改到最后发现"还不如我自己来"，于是彻底放弃了 AI。

这个原则与第 8.4 节"从执行者到管理者"的角色转变完全呼应，但补充了一个关键认知：**管理者的价值不在于"做"，而在于"定义什么是好的"。** 当你告诉 AI "这个实现不够优雅，因为耦合度太高"时，你不需要亲自写出解耦后的版本——你只需要定义"优雅"的标准（内聚、解耦、单一职责），让 AI 自己迭代到符合标准。

**在 Claude Code 中的实践**：

- **Plan Mode 中的角色**：在 Plan Mode（见第 6 章）中，Claude 提出方案，你审查方案。如果方案有问题，你指出问题所在（"这个方案没有考虑并发安全"），然后让 Claude 重新设计——不要直接给出你的设计
- **Code Review 中的角色**：在 Review 环节（见第 7 章），你的审查意见应该描述"问题是什么"和"好的标准是什么"，而不是给出修改后的代码。让 Claude 自己去改——这样它才"学会"了
- **CLAUDE.md 的写法**：CLAUDE.md 应该写的是"什么是对的"（标准、原则、示例），而不是"具体怎么做"（步骤、代码片段）。后者属于 Skill 或具体 prompt，前者属于长期记忆

### 外部视角的总结

汉洋团队在播客结束时给出了一个检验标准：**"你愿意在这个作品上署自己的名字吗？"** 如果答案是"不愿意"，说明你的意志没有灌注到 AI 的生成过程中——你只是让 AI 代笔，而不是让 AI 协助你创作。

这个标准在 Claude Code 中的对应版本是：**"这个代码库在你离开后，团队能维护吗？"** 如果你只是把 AI 当成代笔工具，让它写你不懂的代码，那么答案是否定的。但如果你通过打磨流程、定义标准、审查结果，把你的工程判断灌注到了整个过程中——那么这个代码库就是你的作品，即使很多行是 AI 写的。

## 8.4 从"执行者"到"管理者"的角色转变

徐文浩用他的实际经验描述了这种转变：

### 以前 vs 现在

| | 以前 | 现在 |
|--|------|------|
| 日常 | 自己写代码、逐行 review | 创造环境、看计划、验收结果 |
| 精力分配 | 100% 在写代码 | 分级关注：风险大的重点看，常规的扫一眼 |
| 核心技能 | 编码能力 | 问题定义能力、判断力、流程设计能力 |
| 与 AI 的关系 | 无 | "你是它的辅助，帮它扫清障碍" |

### 四项新日常工作

**1. 创造环境**

给 AI agent 提供良好的工作环境——不是物理环境，而是信息环境：
- 代码库结构清晰（深模块）
- CLAUDE.md 写得完整
- 权限和工具配置正确
- 测试基础设施完善（AI 写完后能自己验证）

**2. 看计划**

不再关注"代码怎么写"，而是关注"方案对不对"：
- 读取 AI 的 Plan Mode 输出
- 评估方案的风险和可行性
- 批准、修改或否决

**3. 验收结果**

不再逐行审查代码，而是验收结果：
- 亲自上手用（QA 是品位把关）
- 检查是否符合需求
- 跑一遍核心用户路径

**4. 分级关注**

不是所有代码都值得你审查：
- 新人写的关键模块 → 重点看
- 常规功能迭代 → 抽查
- 成熟的模板化产出 → 只看测试结果

### 徐文浩的操作实践

> "以前：上午写代码，中下午开会。现在：开会更高效了，因为 AI 已经把代码写完了——会议中的决策可以立即转成执行。"

### 非技术同事也能用

徐文浩让 Claude Code 自动生成图文并茂的 HTML 教程，给非技术同事（如 QA）上手使用：

```
> 为 QA 同事生成一份使用教程：如何用 Claude Code 在本地运行自动化测试。
> 教程需要图文并茂，适合非技术背景。
> 每步都配上截图位置的占位符和详细说明。
```

结果：非技术同事"用得飞起"。

**当前限制**：SaaS 鉴权连接仍需人手工操作（如飞书、Jira 的 OAuth 登录）。这是基础设施层面的缺口，不是 AI 能力的问题。

## 8.5 "滑雪比赛"隐喻：安全边界内敢于放手

徐文浩用滑雪比赛来比喻 AI 开发的节奏：

> "如果 AI 写的代码线上从来没出过 bug，说明你太保守了。你在滑雪比赛里追求零风险，速度必然是最慢的。"

### 零 bug 可能意味着太慢

追求 AI 产出的代码零缺陷，说明你：
- 在大量人工验证（浪费了 AI 的效率优势）
- 不让 AI 做它有能力做的决策（过度控制）
- 产出速度远低于 AI 的实际能力

**正确的风险观**：

在安全边界内（见第 7 章五层闭环），接受一定比例的 bug。这些 bug 会被自动化测试、AI Code Review、沙箱隔离等机制层层过滤。真正能到达生产环境的 bug，远比你想象得少。

### 容错文化的三个层面

**个人层面**：
- 允许 Claude 犯错，把它当成"高级 junior developer"而非"绝不会出错的 oracle"
- 对错误的反应不是"AI 不可靠，我自己来"，而是"这个错误模式该写进 CLAUDE.md 了"

**团队层面**：
- 创造一个"AI 写出 bug 不是丢脸的事"的氛围
- 鼓励分享 AI 出错的案例（这是团队学习的材料）
- 设立"AI 代码事故周报"，总结经验写入规范

**组织层面**（徐文浩三条经验）：

**1. 自己先用出效果**

不要先"论证 AI 适不适合我们团队"，而是自己先用出实际效果，给团队看"我这星期干了什么"。效果是最好的说服工具。

> "你不用跟团队说用 AI 好不好。你展示一下你这周干了 20 件事，他们还在干第 3 件，就有人会来问你怎么做到的。"

**2. 降低门槛**

帮别人配好环境，做简单的一页教程，而不是丢给他们一大堆英文文档。让非技术同事也能用起来，扩散应用范围。

**3. 允许犯错**

带头大方花钱买订阅、买 Token。创造一个"试错不会挨批"的文化。

> "年底预算没用完的部门，买 Claude 订阅是 ROI 最高的投资。"

**反直觉的一点**：

Pro 订阅是固定成本——每个月付固定金额，而不是按用量计费。这种定价模式反而逼你多用：

> "付都付了，不用白不用。多用多赚，越用越发现能用的地方。"

API 按量计费会让你省着用，Pro 固定成本让你放开用。后者更接近"全自动"的节奏。

## 8.6 AI 的局限与人的不可替代性

### 监督悖论：你越依赖 AI，越难判断 AI 对不对

MBZUAI 的论文（Dive into Claude Code）从源码中提取了五个设计价值（见第 1 章 1.2 节），但论文同时使用了一个"评估性镜头"来审视整个架构：**长期人类能力保持**——Claude Code 是否帮助你在长期保持（甚至提升）编程能力，而非让你逐渐丧失它。

答案有些令人不安。论文引用了一系列独立实证研究：

| 研究 | 发现 | 来源 |
|------|------|------|
| 16 名经验丰富的开发者，246 个任务 | AI 工具使开发者**实际慢了 19%**，但**自认为快了 20%** | Becker et al., 2025 |
| 807 个仓库的 Cursor 采用因果分析 | 代码复杂度**增加 40.7%**，初期速度飙升在三个月内消散到基线 | He et al., 2025 |
| AI 辅助组 vs 无 AI 对照组 | AI 辅助组在理解测试中得分**低 17%** | Shen & Tamkin, 2026 |
| 54 名参与者的 EEG 脑电研究 | LLM 用户神经连接减弱，**AI 移除后仍持续** | Kosmyna et al., 2025 |
| 304,000 个 AI 编写 commit，6,275 个仓库 | 约四分之一 AI 引入的问题持续到最新修订版，安全问题持续率更高 | Liu et al., 2026 |

这些研究不是针对 Claude Code 的，而是针对整个"AI 辅助编程"现象。但 Claude Code 是 AI 辅助编程工具中**自主程度最高的之一**——它能自主写代码、执行命令、修改文件——所以这些风险在 Claude Code 中**不是更低，而是更高**。

论文把这个问题命名为**"监督悖论"**：

> Anthropic's own study of 132 engineers and researchers documents a "paradox of supervision" in which overreliance on AI risks atrophying the skills needed to supervise it.

翻译：**过度依赖 AI 编程，可能让你丧失监督 AI 所需的能力。** 而你丧失能力之后，就更无法判断 AI 的输出了——这是一个自我强化的恶性循环。

论文将此定位为一个开放问题，而非 Claude Code 架构已经解决的设计目标：

> "Future systems could treat that sustainability gap as a first-class design problem, not a downstream evaluation metric. The architecture provides limited mechanisms that explicitly preserve long-term human understanding, codebase coherence, or the developer pipeline."

翻译：**未来系统可以把"让人保持能力"当作一等设计问题，而非事后评估指标。当前架构中，几乎没有机制明确支持长期人类理解、代码库一致性或开发者成长。**

### 对抗监督悖论的四个策略

这些数据不是让你放弃 Claude Code，而是提醒你必须**有意识地对抗**能力退化。以下是基于徐文浩、刘小排实践的四个具体策略：

**策略 1：分级关注，而非全放或全查。** 不逐行 review 所有 AI 代码，但也从不完全不看。只看三类：想学习的（了解新技术或新模式的实现）、风险大的（涉及安全、数据、支付的核心路径）、让你困惑的（AI 做了一個你不理解的决策）。徐文浩的做法是："其他东西我就是快速浏览一下，我就合并了。"

**策略 2：自己写最核心的模块。** 不要让 AI 写一切。识别出项目中"如果你不理解就等于丧失了代码库所有权"的核心路径，这些自己写，或至少在 AI 写后逐行理解。对于模板化的、重复的、非核心的代码，才是 AI 的用武之地。

**策略 3：用 QA 保持判断力。** QA 不是点击测试——Matt Pocock 说 QA 是品位把关（见第 6 章）。亲自用你的产品、审查 AI 的设计决策、评估它的架构选择。这是在保持你的审美判断力和工程直觉。

**策略 4：定期做"裸写"练习。** 至少每周有一次关闭 AI，完全自己写代码。可以是小任务——修一个 bug、重构一个模块、写一个算法实现。目的是保持你的神经系统不被"重塑"为完全依赖 AI 的模式——EEG 研究表明这种重塑可能在 AI 移除后仍持续。

### 训练器式用法的五条实践规则

以上四个策略来自徐文浩、刘小排的工程实践。以下五个规则来自另一个视角——**高强度使用 SOTA Coding Agent 后的第一人称反思**。这两条路径指向同一个目标：让 AI 成为你能力的放大器，而不是你能力的替代物。

**规则 1：先预测，再询问。**

不要一上来就把问题丢给 Claude。先写下自己的判断：

- 我认为问题的根因是 A
- 可能的方案是 B
- 最大的风险是 C
- 我预计 Claude 会指出 D

然后再让 Claude 评估。这个顺序的关键在于：**认知负荷必须留在人身上。** 如果你跳过"自己先想"这一步，你只是在消费答案，而不是训练判断力。人的学习发生在"预测错了→发现冲突→修正模型"的过程中，而不是"看到答案→觉得懂了→然后忘了"的过程中。

**规则 2：先拆问题，再让 Claude 扩展。**

不要把一个模糊目标直接丢给 Claude。先自己拆成：

- 目标是什么
- 约束有哪些
- 已知事实
- 未知问题
- 成功标准
- 不接受的方案
- 需要人审的点

这个拆解过程本身就是你在形成判断力。Claude 可以帮你扩展边界、补齐盲区，但**问题框架必须由人建立**。如果问题框架是错的，AI 执行得越高效，离正确结果越远。

**规则 3：强制 review Claude 的过程，而不只是结果。**

不要只看 Claude 最终输出的代码。要看它的思考过程：

- 它是否误解了问题？
- 它是否跳过了关键约束？
- 它是否过度工程化了？
- 它是否在局部正确但整体错误？
- 它是否给出了看似完整但无效的方案？

Claude Code 的 Plan Mode（见第 6 章）正是为此设计的——它强迫 Claude 在动手写代码之前先把方案亮出来，让你 review。但即使在不使用 Plan Mode 的情况下，你也可以要求 Claude 先给出思路再执行。

**规则 4：每次使用后沉淀一个模型。**

一次 Claude Code 使用结束后，至少留下其中一种东西：

| 沉淀物 | 示例 |
|--------|------|
| 一个 checklist | "下次迁移组件时，先检查依赖注入方式" |
| 一个判断原则 | "涉及数据库 schema 的修改必须人工 review" |
| 一个反例 | "Claude 在这个文件里误用了异步模式，下次注意" |
| 一个失败模式 | "当需求涉及第三方 API 时，Claude 经常漏掉错误处理" |
| 一个 SOP | "添加新页面时的标准步骤：路由→组件→测试→文档" |
| 一个可复用模板 | "CRUD 接口的标准 prompt 模板" |

如果一次使用只留下了输出（代码、文档、配置），没有留下人的判断模型，那这次使用大概率只是**消费**，不是**学习**。

**规则 5：把执行外包，不把判断外包。**

这三句话应该成为你的口头禅：

> 把执行外包给 AI，但不要把判断外包给 AI。  
> 把试错加速交给 AI，但不要把理解过程交给 AI。  
> 把候选空间扩展交给 AI，但不要把最终审美和选择交给 AI。

具体在 Claude Code 中的落地：

| 可以外包给 Claude | 必须留在人身上 |
|------------------|---------------|
| 根据明确需求改代码 | 判断这个需求是否值得做 |
| 写测试覆盖 | 判断测试策略是否合理（单元/集成/E2E 的比例） |
| 机械重构 | 判断重构是否引入了行为变更 |
| 生成样板代码 | 判断样板是否过度抽象 |
| 在大代码库里找路径 | 判断找到的路径是否符合架构原则 |
| 并行探索多个实现方案 | 判断哪个方案最好 |

### AI 的能力边界：Bounded vs Large Solution Space

Claude Code 不是万能的。经过几个月高强度使用后，可以把它定位为一个**高吞吐执行器 + 第二视角 + 反复试错器 + 局部专家集合**。它擅长在**有界问题空间**（bounded problem space）里工作，但在**开放解空间**（large solution space）里，人的 taste、直觉、经验和审美才是核心。

**Claude Code 已经很强的部分（有界空间）：**

- 根据明确需求改代码
- 在局部上下文里修 bug
- 写测试
- 做机械重构
- 串接 API
- 生成样板代码
- 按现有风格补齐模块
- 在大代码库里帮人找路径
- 并行探索多个实现方案
- 快速验证假设

这些能力非常有价值。它们让一个人拥有了过去小团队才有的执行吞吐。

**Claude Code 仍然弱的部分（开放空间）：**

- 这个问题是否值得解决
- 需求是否被定义对了
- 架构边界是否合理
- 长期维护成本
- 产品体验里的细微判断
- 隐含约束（法规、合规、团队惯例）
- 代码库里的历史脉络（"为什么三年前要这样设计"）
- 多轮执行中的方向稳定性
- 什么叫"够了"
- 什么叫"漂亮"

也就是说，Claude Code 可以扩展候选空间、加速试错、补齐机械细节，但它**不会自然知道哪一个值得押上去**。判断"哪个方案值得做"、"什么时候该停止"、"这个 tradeoff 是否可接受"——这些是人的领地。

一个更硬的标准来检验你是否在正确的边界上使用 Claude Code：

> 如果一次使用后，只留下了输出，没有留下人的判断模型，那这次使用更接近消费，不是学习。  
> 有效使用 AI 后，人应该变得更强，而不是只拥有更多结果。

可以追问自己：
- 我这次是否形成了一个新的判断标准？
- 我是否知道了一个新的 failure mode？
- 我是否能更快识别同类问题？
- 下次没有 Claude Code，我是否也会比之前更强？

如果答案都是否，说明你需要回到"训练器式用法"。

### AI 目前的真正局限

| 局限 | 说明 | 人的优势 |
|------|------|---------|
| 真实物理世界 | 手机和电脑能触达的虚拟世界，给 SOP 就能自动化。但物理世界（如操作硬件设备、面对面沟通）仍是盲区 | 手、眼、腿、情感 |
| 人际关系判断 | HR、行政、客户关系等需要微妙判断的领域 | 同理心、直觉、政治敏感 |
| 大文件编辑 | 行数变化后 edit 工具的定位会失败 | 对整体结构的直觉理解 |
| 没有真正的"记忆" | 每次 /clear 回到起点，像《记忆碎片》主角 | 持久记忆、经验积累 |

> "一切能 SOP 化的事情，Claude Code 都能做。" ——刘小排

反过来说：**不能 SOP 化的事情，才是人的核心价值所在。**

### 人的不可替代性

在 AI 时代，人的价值在**八个维度**增强而非减弱。前四个来自工程实践，后四个来自高强度使用 Agent 后的反思——它们共同构成了"人在 AI 时代的核心竞争力模型"。

**1. 判断力（Judgment）**

AI 可以列出所有选项，但不能替你决策。判断"什么是好的"、"什么是重要的"——这是人的核心能力。

更深一层：判断力不仅是做选择，还包括**在不确定条件下做选择并承担后果**。AI 可以生成 10 个实现方案，每个都有利弊说明，但它不会替你承担"选错方案导致项目延期三个月"的后果。这种"承担"本身就是一种不可替代的价值。

**2. 品味（Taste）**

AI 可以写出正确的代码，但不能判断"这个 API 设计得好不好"。深模块 vs 浅模块、好的抽象 vs 过度抽象——这些需要审美判断。

更关键的是：**品味不仅决定"什么是好的"，还决定"每一轮迭代应该往哪里推"。** AI 可以执行迭代（改代码、跑测试、再改），但它不知道这一轮应该优化可读性还是优化性能，不知道应该增加抽象还是减少抽象。方向的把控是人的领地。

品味来自长期暴露、比较、失败、模仿和自我修正——它不是 prompt 能调出来的参数。

**3. 上下文（Context）与压缩能力（Compression）**

AI 只能理解你给它的上下文。你脑子里的用户故事、团队动态、业务压力——这些 AI 不知道。把正确的上下文喂给 AI，本身就是一个技能。

但还有一层被忽视的能力：**把大量信息压缩成原则、模型、判断标准的能力。** 你读了 10 个类似的 bug 报告、3 次代码审查反馈、2 次用户投诉，你能提炼出一条"这类问题通常是因为 X"的原则。AI 可以帮你整理信息，但压缩成"可迁移的判断标准"需要人的抽象能力。

**4. 责任（Responsibility）与审查能力（Review Ability）**

AI 不会为 bug 负责。最终签字的还是你。这意味着你必须保持足够的理解力，判断 AI 的方案是否安全。

审查不是简单看一眼——是**能发现 AI 输出里的漏洞、幻觉、过度工程和边界错误**。这需要你：
- 知道系统里有哪些隐含约束（法规、合规、团队惯例）
- 知道代码库里的历史脉络（"为什么三年前要这样设计"）
- 能判断"局部正确但整体错误"的方案

**5. 问题定义能力（Problem Framing）**

AI 擅长解决被定义好的问题，但它不擅长定义问题。

真正困难的不是"怎么实现登录功能"，而是"用户需要的到底是登录功能还是身份验证功能？""要不要支持社交登录？""匿名浏览和强制登录的边界在哪里？"——这些问题的答案决定了后续所有实现的方向。如果问题框架是错的，AI 执行得越高效，离正确结果越远。

**6. 过程控制（Process Control）**

能管理 Claude Code 的"跌跌撞撞"，而不是被它的过程带跑。

Claude Code 在多轮执行中可能会偏离方向：它可能在第 5 轮改了一个与当前任务无关的文件，可能在第 8 轮引入了一个看似合理但会破坏兼容性的改动。你需要在过程中不断校准方向——不是等它跑完了再全盘否决，而是在关键节点介入、纠正、重新设定约束。

**7. 有品位的迭代（Tasteful Iteration）**

不只是迭代，而是知道每一轮应该往哪里推。

AI 可以执行"改代码→跑测试→修 bug→再跑测试"的循环，但它不知道这一轮的目标应该是"让代码更易读"还是"让性能提升 20%"还是"减少一个不必要的依赖"。迭代方向的判断力——什么时候该停、什么时候该推、什么时候该换方向——是人的能力。

**8. 亲历学习（Embodied Learning）**

自己亲自经历理解、犯错、修正、吸收，而不是只看结果。

最有效的学习发生在这些时刻：自己先预测→发现预测错了→和反馈发生冲突→修正模型→抽象成原则→下一次迁移使用。如果 Claude Code 把这些全部替人做了，人只是在消费认知结果，而不是训练认知能力。

这也是为什么第 8.5 节的"裸写练习"如此重要——它强制你保留"亲历"的摩擦。不是为了低效，而是为了学习。

### 鸦片式用法 vs 训练器式用法：你正在用哪种方式？

以上所有策略和规则可以归结为两种根本不同的使用方式。对照下表，诚实评估自己目前更接近哪一种：

| 维度 | 鸦片式用法 | 训练器式用法 |
|------|-----------|-------------|
| **心态** | "我不想想，你替我想" | "我先给出判断，你挑错/扩展/补盲区" |
| **提问方式** | 直接丢模糊目标 | 先拆解问题框架，再让 AI 填充 |
| **对输出的态度** | 接收即用 | 审查过程，质疑假设，验证边界 |
| **每次使用后的沉淀** | 只有输出（代码/文档） | 留下 checklist / 原则 / 反例 / 失败模式 |
| **能力曲线** | 表面高效，实际判断力停滞或退化 | 初期略慢，长期能力持续上升 |
| **没有 AI 时的状态** | 比用 AI 之前更弱 | 比用 AI 之前更强 |
| **典型对话** | "帮我写个登录功能" | "我认为登录功能应该支持邮箱+手机号双因子，约束是不引入新依赖，你先评估这个方案的风险" |

**关键检验标准**：

> 如果一次 Claude Code 使用结束后，只留下了输出，没有留下人的判断模型，那这次使用大概率只是**消费**，不是**学习**。
>
> 有效使用 AI 后，人应该变得更强，而不是只拥有更多结果。

每次使用完后问自己四个问题：

1. 我这次是否形成了一个新的判断标准？
2. 我是否知道了一个新的 failure mode？
3. 我是否能更快识别同类问题？
4. 下次没有 Claude Code，我是否也会比之前更强？

如果答案多数是"否"，说明你正在滑向鸦片式用法。回到训练器式用法的起点：**先预测，再询问。**

## 8.7 实用习惯清单

从以上内容提炼的日常实操清单：

| 习惯 | 具体做法 | 为什么 |
|------|---------|--------|
| 每天开始 | 检查 CLAUDE.md 是否需要更新（昨天发现的新问题） | 防止 Doc Rot，让 AI 今天不再犯同样的错 |
| 遇到重复操作 | 主动思考"这个能不能变成 prompt 模板" | 从执行者思维切换到流程设计师思维 |
| QA 发现问题 | 绝不自己改，写 issue 丢回看板 | 保持工作流不中断，同时训练 AI 的自我修复能力 |
| 周末回顾 | 看本周有哪些流程可以优化 | 持续打磨流程，每次优化都会放大到后续 100 次执行 |
| 分享案例 | 把"AI 犯了一个有意思的错"分享给团队 | 建立容错文化，同时把个体经验变成团队知识 |
| 分级对待 | 高风险代码亲自审查，常规产出抽查 | 把有限的注意力花在最重要的地方 |

**终极建议**：

> 忘掉"我是程序员"这个身份标签。你现在是一个**系统的设计者**，这个系统包含了一个或多个 AI agent。你的工作不再是写代码，而是让这个系统的产出质量越来越高、速度越来越快、人力投入越来越少。
