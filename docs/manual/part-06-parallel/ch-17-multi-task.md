# 第17章 多任务并行

> "我怎么知道后台的 Claude 任务跑到哪了？""能让 Claude 同时处理多个不相关的任务吗？""子代理和主会话到底是什么关系？"

第15章讲 MCP 连接外部系统，第16章讲插件如何打包扩展能力。但安装扩展只是第一步——当你同时运行多个任务、派发子代理处理复杂工作时，你需要看到后台在发生什么、主会话和子代理如何协作、以及它们之间的边界在哪里。

本章讲 Claude Code 的多任务并行机制：从可视化后台任务，到子代理调度，再到权限隔离。这些能力是第18章 Worktree 隔离和第19章四角色流水线的操作层面基础。

---

## 17.1 Agent View：可视化后台多任务管理

### 问题：我派了一个子代理去后台运行，怎么知道它跑到哪了？

**背景**：当你使用子代理处理长任务时（例如"探索整个代码库并找出所有废弃函数"），主会话不需要阻塞等待。子代理在后台运行，你可以继续和主会话交互。但这也带来了一个问题：**后台任务的进度、输出、状态对你是不可见的**，除非它主动返回结果。

**为什么是问题**：人类的工作记忆有限。如果你同时派了三个子代理分别做"探索代码库""运行测试""查找文档"，几分钟后你可能忘了哪个是哪个、哪个已经失败、哪个还在跑。没有可视化界面，并行任务的管理成本会指数增长。

**Agent View 的解决机制**：Claude Code 提供了 `claude agents` 命令，展示所有正在运行和已完成的子代理的**仪表盘视图**。

**结论**：Agent View 是多任务并行的"控制塔"——你不用记住每个后台任务的状态，看仪表盘就行。

**Agent View 展示的信息**：

| 字段 | 含义 | 用途 |
|------|------|------|
| **ID** | 子代理唯一标识 | 用于 `attach`、`logs`、`stop` |
| **Name/Type** | 子代理类型（Explore/Plan/Verification 等） | 快速识别任务性质 |
| **Status** | running / succeeded / failed / stopped | 判断是否需要干预 |
| **Progress** | 当前步骤 / 总步骤（部分类型支持） | 估算剩余时间 |
| **Output Summary** | 最近输出的摘要 | 不进入详情也能了解进展 |
| **Start Time** | 启动时间 | 识别僵尸任务 |

**怎么做**：

```bash
# 查看所有子代理状态
claude agents

# JSON 格式输出（用于脚本集成）
claude agents --json

# 切换到特定子代理的会话继续交互
claude attach <agent-id>

# 查看子代理的完整输出日志
claude logs <agent-id>
```

**随时切换前台继续交互**：Agent View 不只是"查看"——你可以随时把一个后台运行的子代理切换到前台，继续和它交互。这让你能从"监督模式"切换到"协作模式"，无需重启任务。

## 17.2 后台运行：`claude --bg`

### 问题：我能一次开十几个任务让它们自己跑吗？

**背景**：Agent View 解决的是"可视化"问题，但前提是任务已经在运行。传统的交互式 REPL 模式下，你一次只能和一个 Claude 会话交互。如果你需要同时处理多个独立任务（例如"审查 PR #123""重构 utils 模块""更新文档"），来回切换会话非常低效。

**为什么需要后台运行**：人类可以并行监督多个任务，但传统 CLI  forcing 你串行处理。后台运行让 Claude Code 从"一对一对话"变成"一个主会话 + 多个后台 worker"的模式。

**Supervisor 进程管理**：Claude Code 使用 supervisor 进程管理后台任务，生命周期如下：

```
Start    → 用户执行 claude --bg "task description"
  ↓
Monitor  → supervisor 跟踪任务状态、输出、资源使用
  ↓
Attach   → 用户可以随时 attach 到后台任务查看或交互
  ↓
Respawn  → 如果任务异常退出（非用户停止），可选自动重启
  ↓
Stop     → 任务完成、用户手动停止、或失败达到重试上限
```

**结论**：后台运行让你能把 Claude 当作"可并行化的工作队列"使用——提交一批任务，让它们各自运行，你只需监督结果。

**启动后台任务的命令**：

```bash
# 启动后台任务（自然语言描述）
claude --bg "审查 src/auth/ 目录的所有代码，找出潜在安全问题"

# 启动多个后台任务
claude --bg "运行所有测试并生成覆盖率报告"
claude --bg "查找项目中所有 TODO 注释并按文件分组"
claude --bg "对比当前分支和 main 的 API 差异"

# 查看所有后台任务
claude agents

# 查看特定任务的输出
claude logs <task-id>
```

**后台任务与前台会话的区别**：

| | 前台会话 | 后台任务 |
|--|---------|---------|
| **交互方式** | 实时对话 | 提交后自行运行 |
| **上下文共享** | 与主会话共享上下文 | 独立上下文（可配置） |
| **失败处理** | 立即显示，用户即时干预 | supervisor 记录，用户稍后查看 |
| **适用场景** | 需要频繁澄清的复杂任务 | 目标明确、可独立完成的子任务 |
| **资源占用** | 占用当前终端 | 后台进程，不影响前台工作 |

**怎么做**：
- 目标明确、可独立验证的子任务 → 用 `--bg` 后台运行（如运行测试、查找代码、生成报告）
- 需要频繁交互、澄清需求的任务 → 保持前台会话
- 批量处理（如"审查所有 PR"）→ 开多个 `--bg` 任务并行处理

## 17.3 Agent 工具：子代理调度机制

### 问题：主会话是怎么"派活"给子代理的？子代理知道主会话让它做什么吗？

**背景**：第17.1~17.2 讲的是用户手动启动并行任务。但 Claude Code 更强大的能力在于**主会话自动派发子代理**——当主会话判断某个任务适合独立执行时，它会调用 Agent 工具创建一个子代理。

**为什么需要自动调度**：用户手动管理并行任务有认知负担。Agent 工具让主会话自己决定"这件事我该自己做，还是派个专业代理去做"。例如，当你说"帮我探索这个陌生代码库"，主会话可能判断这是一个独立的探索任务，自动调用 Explore agent，自己继续处理其他事情。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 8.1）通过源码逆向还原了 Agent 工具的完整输入 schema：

**Agent Tool 的结构化输入**：

```typescript
{
  // 委派指令：主会话告诉子代理要做什么
  "delegation_prompt": "探索 src/ 目录的组件结构，找出所有使用了废弃 API 的文件",

  // 子代理类型：使用哪种内置或自定义代理
  "subagent_type": "Explore",

  // 隔离模式：in-process / worktree / remote
  "isolation": "worktree",

  // 工作目录：子代理的 CWD
  "cwd": ".claude/worktrees/explore-task",

  // 权限覆盖：子代理的权限可独立于主会话配置
  "permission_mode": "restricted",
  "allowed_tools": ["Read", "Grep", "Glob"],

  // 后台运行标志
  "run_in_background": true,

  // 其他配置
  "model": "claude-sonnet-4-6",
  "max_turns": 50,
  "skills": ["code-review"]
}
```

**Feature-gated fields**：上述部分字段受 feature gate 控制，不是所有用户都能使用全部隔离选项。这反映了 Claude Code 对子代理能力的渐进式开放策略——核心功能（subagent_type、delegation_prompt）默认可用，高级隔离选项逐步 rollout。

**结论**：Agent 工具是主会话和子代理之间的"委派协议"。主会话通过结构化输入精确控制子代理的任务范围、隔离级别、权限边界，而不是简单地"fork 一个自己"。

**Agent Tool 的执行流程**：

```
主会话判断需要子代理
  ↓
组装 Agent Tool 输入（delegation_prompt + subagent_type + isolation + permissions）
  ↓
创建子代理上下文（新上下文窗口或新 worktree）
  ↓
子代理独立运行，处理 delegation_prompt
  ↓
子代理完成 → 返回结构化结果（summary + artifacts）
  ↓
主会话接收摘要，决定下一步
```

**怎么做**：
- 通常情况下你不需要手动调用 Agent 工具——主会话会自动判断何时派发子代理
- 如果你想强制使用特定类型的子代理 → 在 prompt 中明确要求（如"派 Explore agent 去查找..."）
- 如果你需要控制子代理的权限 → 通过 AGENT.md 配置（见 17.5）

## 17.4 六种内置子代理类型

### 问题：有哪些现成的子代理类型？它们各适合什么场景？

**背景**：Claude Code 不是让每个子代理都从零配置，而是提供了六种**内置子代理类型**。每种类型有预设的工具集、权限模型和行为模式，针对特定场景优化。

**为什么需要内置类型**：如果每次派子代理都要手动配置工具列表、权限、隔离模式，使用成本太高。内置类型是"开箱即用的专业角色"——就像团队里有专门的测试工程师、架构师、代码审查员，你不需要每次告诉他们"你可以读代码但不能改生产配置"。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 8.1）定义了六种内置类型：

| 类型 | 核心能力 | 默认工具限制 | 典型任务 | 隔离模式 |
|------|---------|-------------|---------|---------|
| **Explore** | 只读搜索与探索 | **Write/Edit 在 deny-list** | 代码库结构分析、查找引用、依赖梳理 | worktree |
| **Plan** | 创建结构化计划 | 标准权限模型 | 需求拆解、制定实施步骤、估算工作量 | in-process |
| **General-purpose** | 通用能力 | 标准权限模型 | 实现功能、修复 bug、重构代码 | worktree |
| **Claude Code Guide** | 文档与上手辅助 | 只读 + 特定文档工具 | 回答使用问题、解释功能、引导上手 | in-process |
| **Verification** | 运行验证检查 | 受限执行权限 | 运行测试、执行 lint、检查类型 | worktree |
| **Statusline-setup** | 终端状态栏配置 | 高度受限 | 配置终端状态栏显示、优化提示格式 | in-process |

**各类型详解**：

**Explore**：最安全的子代理类型。它的 write 和 edit 工具默认在 deny-list 中，意味着它只能**读取**代码库而不能修改。这让它适合探索陌生代码、查找所有使用了某 API 的文件、分析依赖关系——你不用担心它意外修改了代码。

**Plan**：专门用于创建结构化实施计划。它会分析需求，输出步骤化的执行方案，但不直接执行。Plan agent 通常在 in-process 模式下运行（不需要文件系统隔离），因为它主要产出的是文本计划而非文件修改。

**General-purpose**：默认的"全能型"子代理。没有特殊的工具限制，可以读写文件、执行命令、调用工具。大多数"实现功能""修复 bug"类的任务会派这种代理。

**Claude Code Guide**：专门回答 Claude Code 使用问题的代理。它读取官方文档和内置帮助内容，帮助用户理解功能、解决使用困惑。

**Verification**：专门运行验证检查的代理。它执行测试、lint、类型检查等验证任务，返回通过/失败结果和相关日志。通常与 Implementer 代理配合——一个写代码，一个验证。

**Statusline-setup**：高度特化的代理，用于配置终端状态栏（如自定义提示符、显示 Git 状态等）。权限非常受限，只操作配置相关文件。

**结论**：内置子代理类型是"预设好的专业角色"。选择合适的类型，相当于给任务匹配了最合适的专家，同时获得了预设的安全边界。

**怎么做**：
- 只需要读代码、找东西 → `Explore`（最安全）
- 需要制定方案但不执行 → `Plan`
- 一般的代码实现/修复 → `General-purpose`
- 需要验证代码质量 → `Verification`
- 回答 Claude Code 使用问题 → `Claude Code Guide`

## 17.5 自定义子代理：`claude/agents/*.md`

### 问题：内置类型不够用，我能自己定义一个专用子代理吗？

**背景**：六种内置类型覆盖了常见场景，但实际工作中你可能需要更专门化的代理。例如：
- 一个专门做安全审计的代理（只读 + 安全规则注入）
- 一个专门做性能分析的代理（特定工具集 + 性能阈值配置）
- 一个专门做 i18n 翻译的代理（特定文件路径 + 翻译规范）

**为什么需要自定义**：不同团队、不同项目有特定的工作流和规范。自定义子代理让你能把这些规范"固化"成可复用的代理配置，而不是每次都在 prompt 里重复说明。

**自定义子代理的配置方式**：在项目或用户目录下创建 `claude/agents/*.md` 文件，使用 YAML frontmatter + Markdown 正文：

```markdown
---
name: security-auditor
description: 安全审计专用代理，扫描代码中的潜在安全问题
type: Explore  # 基于哪个内置类型扩展
model: claude-opus-4-8
effort: high

# 工具配置
tools:
  - Read
  - Grep
  - Glob
disallowedTools:
  - Write
  - Edit
  - Bash

# 权限
permissionMode: restricted

# MCP 和技能
mcpServers:
  - security-scanner
skills:
  - owasp-checklist

# 上下文
memory: true
maxTurns: 100

# 运行配置
background: true
isolation: worktree
---

# 安全审计规范

## 检查清单
- [ ] SQL 注入风险（字符串拼接 SQL）
- [ ] XSS 风险（未转义的用户输入输出）
- [ ] 敏感信息泄露（密钥、token 硬编码）
- [ ] 不安全的反序列化
- [ ] 权限绕过漏洞

## 报告格式
对每个发现的问题，输出：
1. 文件路径和行号
2. 风险等级（Critical / High / Medium / Low）
3. 问题描述
4. 修复建议
```

**Agent Tool vs Skill Tool 的本质区别**：

这是自定义子代理时最容易混淆的概念。

| | **Agent Tool（子代理）** | **Skill Tool（技能）** |
|--|------------------------|----------------------|
| **上下文** | 创建**新**上下文窗口 | 注入**当前**上下文 |
| **执行方式** | 独立运行，可后台执行 | 在当前会话内执行 |
| **隔离性** | 可配置隔离（worktree/in-process） | 无隔离，与主会话共享一切 |
| **适用场景** | 独立任务、需要隔离的工作 | 当前任务内的特定步骤 |
| **返回方式** | summary-only 返回摘要 | 结果直接进入当前上下文 |

**为什么这个区别重要**：
- 如果你需要**并行处理**一个独立任务 → 用 Agent（创建新上下文，不污染主会话）
- 如果你只是需要**复用一段 prompt**来完成当前任务的某一步 → 用 Skill（注入当前上下文，共享状态）

**结论**：自定义子代理让你能创建"团队专用角色"——把特定领域的规范、工具集、权限模型封装成可复用的代理定义。

**怎么做**：
- 定义 AGENT.md → 放在 `.claude/agents/` 或 `~/.claude/agents/`
- 主会话调用时引用代理名称 → "派 `security-auditor` agent 扫描 `src/api/`"
- 定期审查自定义代理配置 → 删除不再使用的，更新过时的工具列表

## 17.6 给每个 Agent 分配专精领域，prompt 单独打磨

### 问题：自定义了代理，但效果还是不好，为什么？

**背景**：创建自定义代理只是第一步。就像招聘了一个新员工，给了他职位描述，但如果职位描述模糊、目标不清晰，他还是做不好。

**为什么是问题**：很多用户创建自定义代理时，只配置了 `tools` 和 `disallowedTools`，但忽略了最关键的部分——**delegation prompt 的质量**。子代理接收到的 delegation prompt 决定了它理解任务的方式、输出的格式、关注的重点。

**专精领域的分配原则**：

| 代理类型 | 专精领域 | Prompt 打磨重点 |
|---------|---------|---------------|
| Explore | 代码库理解 | 明确搜索范围、输出格式（列表/树状图/依赖图） |
| Plan | 需求拆解 | 明确计划粒度、依赖关系表达方式、验收标准 |
| Verification | 质量检查 | 明确通过/失败标准、输出格式（摘要 vs 详细日志） |
| Security Auditor | 安全扫描 | 明确检查清单、风险分级标准、 CWE 编号映射 |
| Performance Analyst | 性能优化 | 明确基准指标、瓶颈定义、优化方向优先级 |

**Prompt 打磨的具体方法**：

1. **明确输出格式**：不要只说"检查一下"，要说"输出 Markdown 表格，列：文件路径、行号、问题描述、风险等级、修复建议"

2. **提供判断标准**：不要只说"找 bug"，要说"bug 定义为：会导致程序崩溃、数据丢失、或安全漏洞的问题。样式问题不算 bug"

3. **限定范围**：不要只说"审查代码"，要说"只审查 `src/api/` 目录下的 TypeScript 文件，忽略测试文件和第三方库"

4. **给示例**：在 AGENT.md 中提供 1~2 个输入/输出示例，让子代理理解期望的质量标准

**结论**：子代理的效果 = 代理配置（工具/权限） × delegation prompt 质量。后者往往被忽视，但影响更大。

**怎么做**：
- 第一次使用自定义代理后，审查它的输出 → 如果偏离预期，修改 AGENT.md 中的 prompt
- 第二次纠正同类型问题 → 把修正写进 AGENT.md（呼应第10章和第12章的"纠正两次写进规则"）
- 不要期望一次写出完美的代理配置 → 迭代打磨，像培训新员工一样

## 17.7 子代理与主会话的隔离

### 问题：子代理能做什么？它会不会越权访问我的文件或执行危险命令？

**背景**：第17.3~17.5 讲了子代理的调度和类型，但没有回答一个关键安全问题：子代理的权限边界在哪里？如果主会话有权限执行 `Bash` 命令，子代理是否自动继承？如果一个子代理被攻击者控制，它能造成多大损害？

**为什么需要权限隔离**：子代理是独立运行的实体，可能处理不可信输入（例如 Explore agent 读取用户提交的代码）。如果子代理拥有和主会话相同的权限，一个恶意文件或 prompt 注入就可能通过子代理执行危险操作。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 8.2）定义了**权限覆盖逻辑（Permission Override Cascade）**：

**两阶段权限模型**：

```
阶段一：SDK-level permissions（全局硬边界）
  └── 由启动参数或托管设置定义，如 --allowedTools
  └── 应用于所有 agent，包括子代理，不可被覆盖
  
阶段二：Session-level rules（会话级规则）
  └── 由当前会话的权限配置定义
  └── 子代理可继承，也可被 AgentTool 的 permission overrides 替换
```

**关键规则**：
1. **SDK-level 权限是硬边界**：如果 `--allowedTools` 排除了 `Bash`，任何子代理都不能使用 `Bash`，即使它的 AGENT.md 显式允许
2. **Session-level 规则可被子代理覆盖**：如果主会话允许 `Write`，但子代理的 `permissionMode: restricted` + `allowed_tools: [Read, Grep]`，则子代理只能读不能写
3. **继承规则**：如果 AgentTool 未提供 `allowedTools`，子代理继承父级 session-level rules

**异步代理的权限提示处理**

当子代理在后台运行时，如果需要用户审批（例如执行一个被限制的工具），权限提示的处理方式取决于 `canShowPermissionPrompts` 设置：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **Bubble** | 子代理的权限提示上浮到主会话，由用户统一审批 | 前台派发的子代理，用户在线监督 |
| **Default Cascade** | 子代理独立处理自己的权限提示（如果用户已 attach）| 用户已 attach 到子代理会话 |
| **Silent Deny** | 无用户在线时，权限请求自动拒绝 | 纯后台任务，无监督 |

**为什么是这种设计**：异步代理的权限处理需要在**安全性**和**可用性**之间平衡。如果所有后台代理的权限提示都静默拒绝，很多任务会无法完成；如果都自动允许，又会有安全风险。Bubble 模式让用户能在主会话统一审批，无需切换上下文。

**结论**：子代理的权限遵循"最小权限原则"——默认受限，显式授权，硬边界不可突破。主会话通过 AgentTool 的 `permission_mode`、`allowed_tools`、`disallowedTools` 精确控制子代理的能力范围。

**怎么做**：

1. **默认使用受限模式**：自定义代理的 `permissionMode` 默认设为 `restricted`，只显式允许需要的工具

2. **Explore 代理永远不给写权限**：只读探索任务显式设置 `disallowedTools: [Write, Edit, Bash]`

3. **审查后台代理的权限**：启动后台任务前，确认它的工具列表是否合理——"一个只查找代码的代理为什么需要 Bash 权限？"

4. **利用 SDK-level 硬边界**：对于不可信环境（如审查外部 PR），用 `--allowedTools` 设置全局硬边界，确保子代理即使被攻破也无法越权

```bash
# 示例：在受限模式下启动 Claude Code，所有子代理都不能写文件或执行命令
claude --allowedTools Read,Grep,Glob
```
