# 第20章 权限与审批

> "Claude 能自动执行命令，我怎么知道它不会删了我的数据库？""为什么有些操作要问我，有些却不问？""我能完全信任自动模式吗？"

第19章的四角色流水线提到了关键节点保留人工审批。但审批机制本身是怎么工作的？系统怎么决定一个操作需要审批、自动允许、还是拒绝？

本章深入 Claude Code 的权限与审批系统——不是告诉你"怎么关闭权限提示"，而是解释**系统为什么这样设计**、**每种权限模式意味着什么**、**deny-first 规则如何评估**、以及**自动模式分类器是怎么做决策的**。理解这些，你才能正确配置权限，而不是盲目绕过安全机制。

---

## 20.1 五种价值观驱动的安全设计

### 问题：为什么 Claude Code 的权限系统这么"啰嗦"？能不能一键关闭所有提示？

**背景**：Claude Code 的权限系统可能是新用户最先注意到的"摩擦点"——每次执行 Bash 命令、修改文件、访问外部系统，系统都会提示你确认。很多用户的第一反应是"怎么关掉这些提示"。

**为什么是问题**：如果权限提示只是"为了安全而安全"的形式主义，那它确实应该被关掉。但 Claude Code 的权限系统不是事后补丁，而是**架构的核心设计原则**。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 2.1）明确了五种驱动安全设计的价值观：

| 价值观 | 含义 | 对权限系统的影响 |
|--------|------|----------------|
| **Human Decision Authority** | 人的最终决策权不可被系统绕过 | 关键操作必须经人确认，系统不能替人做不可逆决定 |
| **Safety, Security, and Privacy** | 即使人疏忽，系统也要提供保护 | 默认拒绝可疑操作，不依赖用户每次都能正确判断 |
| **Reliable Execution** | 忠实执行人的意图，而非字面指令 | 权限系统需要理解"用户真正想做什么"，而非机械匹配 |
| **Usability** | 安全不应以牺牲可用性为代价 | 93% 的操作应该无缝通过，只有真正可疑的才提示 |
| **Transparency** | 用户必须理解系统为什么做某个决定 | 权限提示要说明理由，不是神秘的黑箱 |

**93% 批准率的含义**：

*Dive into Claude Code* 引用了内部数据：Claude Code 的权限提示批准率高达 **93%**。这不是说权限系统"没用"——恰恰相反，它说明系统成功地把提示集中在真正需要关注的 7% 操作上。

**为什么是 93% 而非 100%**：如果批准率是 100%，说明权限系统没有过滤掉任何风险操作；如果低于 90%，说明系统过于敏感，用户在无意义的提示上疲劳。

**为什么不是"重构边界设计"而非"增加警告"**：

传统安全系统的做法是"允许所有操作，但在危险操作前加警告"。Claude Code 的做法是"重新设计边界，让危险操作默认无法到达"——通过 deny-first 规则、预过滤、沙箱等机制，在操作执行前就阻止风险，而不是在最后一秒弹出警告。

**结论**：Claude Code 的权限系统不是"烦人的提示框"，而是围绕五种价值观构建的**分层防御架构**。93% 的批准率证明它在安全和可用性之间找到了平衡。

**怎么做**：
- 不要试图"一键关闭所有权限提示"——你是在拆除安全架构
- 如果你发现某个操作被频繁提示 → 检查 deny rules 是否过于宽泛，或考虑切换到更宽松的权限模式（见 20.2）
- 如果你发现某个操作应该被提示但系统没有提示 → 这是 bug，应该报告

## 20.2 七种权限模式

### 问题：Claude Code 有哪些权限模式？我该用哪个？

**背景**：Claude Code 不是"要么全开要么全关"的二元权限系统。它提供了**七种权限模式**，覆盖从"每个操作都问我"到"完全自动执行"的完整光谱。

**为什么是问题**：选择错误的权限模式会导致两种极端：模式太严格，你在无意义的提示上浪费时间；模式太宽松，系统可能在危险操作上自动执行而不通知你。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 5.1）定义了七种权限模式：

| 模式 | 行为 | 适用场景 | 风险等级 |
|------|------|---------|---------|
| **plan** | 每个工具调用都需要显式批准 | 完全不可信环境、首次使用 | 最低 |
| **default** | 标准审批流，新/危险操作提示 | 日常开发（推荐） | 低 |
| **acceptEdits** | 自动接受文件编辑，其他仍需审批 | 信任当前任务、频繁编辑 | 中低 |
| **auto** | 自动模式分类器判断（见 20.5） | 标准化重复任务 | 中 |
| **dontAsk** | 不询问，直接执行（仍受 deny rules 限制） | 完全信任的封闭环境 | 中高 |
| **bypassPermissions** | 绕过权限系统（内部/调试用） | 开发调试、紧急恢复 | 最高 |
| **bubble** | 内部模式，权限请求上浮到父会话 | 子代理、后台任务 | 内部 |

**Graduated Autonomy Spectrum（渐进自治光谱）**：

```
plan → default → acceptEdits → auto → dontAsk → bypassPermissions
 ↑        ↑           ↑          ↑        ↑            ↑
最严格   日常默认     信任编辑    智能自动   全信任      完全绕过
人最累   平衡         人省一点   人更省    人几乎不介入  无保护
```

**五种外部模式 vs 两种内部模式**：

- **外部模式**（用户可配置）：plan、default、acceptEdits、auto、dontAsk
- **内部/条件模式**（系统内部使用）：bubble（子代理权限上浮）、bypassPermissions（调试/紧急）

**为什么需要这么多种**：

单一模式无法覆盖所有场景。开发新项目时需要严格模式（plan/default），维护成熟项目时可以用自动模式（auto），紧急修复时可能需要 dontAsk。渐进自治让用户能在安全和效率之间滑动选择。

**结论**：权限模式是"信任程度的调节器"。从 default 开始，随着对任务和环境的信任增加，逐步放宽。

**怎么做**：
- **新项目/陌生代码库** → `plan` 或 `default`
- **日常开发/已建立信任的代码库** → `default`（推荐）
- **大量重复性编辑任务** → `acceptEdits`
- **标准化流水线（如四角色模型的 Implementer）** → `auto`
- **完全封闭的测试环境** → `dontAsk`
- **永远不要**在生产环境用 `bypassPermissions`

## 20.3 deny-first 规则评估

### 问题：我配置了 allow 规则，为什么 deny 规则还是生效？

**背景**：Claude Code 的权限规则有两种：allow（允许）和 deny（拒绝）。用户可能期望"更具体的规则覆盖更宽泛的规则"——例如先写 `deny all shell`，再写 `allow npm test`，认为后者应该覆盖前者。

**为什么是问题**：如果用户不理解 deny-first 原则，会觉得权限系统的行为"不合理""有 bug"。但实际上这是经过深思熟虑的安全设计。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 5.1）的 deny-first 规则评估：

**核心原则：deny 始终优先于 allow，无论具体程度**

```
规则示例：
  deny: "bash"           ← 拒绝所有 Bash 命令
  allow: "bash(prefix:npm)"  ← 允许 npm 相关的 Bash 命令

 deny-first 的结果：npm test 仍然被拒绝
 原因：deny 优先，allow 不能覆盖 deny
```

**为什么 deny 优先于 allow**：

这是安全系统的经典设计（Fail-Safe Defaults）。如果 allow 能覆盖 deny，攻击者只需找到一个 allow 规则就能绕过所有 deny。例如：
- 你配置了 `deny all database operations` 保护生产数据库
- 如果 allow 能覆盖 deny，某个 Skill 或 MCP 中的 `allow query database` 就会意外开放访问
- deny-first 确保：除非你显式移除 deny 规则，否则保护始终生效

**Tool-level 匹配 vs Content-level 匹配**：

Claude Code 支持两种规则匹配粒度：

| 匹配类型 | 语法 | 匹配对象 | 示例 |
|---------|------|---------|------|
| **Tool-level** | `bash` | 工具名称 | `deny: "bash"` 拒绝所有 Bash 调用 |
| **Content-level** | `bash(prefix:npm)` | 工具名称 + 内容前缀 | `deny: "bash(prefix:sudo)"` 拒绝 sudo 命令 |

**Content-level 匹配的实现**：系统在执行 Bash 命令前，检查命令字符串的前缀是否匹配规则。这让你能精细控制"允许 npm，但禁止 sudo"，而不需要完全禁止 Bash。

**结论**：deny-first 是安全系统的基石设计。它不是 bug，是防止"允许规则意外覆盖保护规则"的核心机制。

**怎么做**：
- 不要试图用 allow 覆盖 deny → 如果你需要允许某个被 deny 的操作，**修改 deny 规则**或将其细化
- 使用 content-level 匹配实现精细控制 → `deny: "bash(prefix:rm -rf /)"` 而非 `deny: "bash"`
- 定期审计 deny rules → 确保它们仍然必要，不过度宽泛

## 20.4 完整授权流水线（四个阶段）

### 问题：当我执行一个操作时，系统内部到底发生了什么？

**背景**：用户看到的权限提示只是整个授权流水线的"输出端"。理解流水线的完整过程，有助于你判断"为什么这个操作被提示了""哪个环节阻止了它"。

**为什么是问题**：如果只看到最终结果（"被拒绝了"），你无法判断是规则配错了、hook 拦截了、还是分类器误判了。调试权限问题需要理解流水线的每个阶段。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 5.2）定义了四阶段授权流水线：

```
阶段一：预过滤（Pre-filtering）
  └── filterToolsByDenyRules()
  └── 在工具池组装阶段就移除被 deny rules 禁止的工具
  └── 被预过滤移除的工具：对模型不可见，模型不会尝试调用

阶段二：PreToolUse Hook
  └── 在工具即将被调用前执行
  └── 可返回：permissionDecision（允许/拒绝）、updatedInput（修改输入）
  └── 用于：动态权限判断、输入消毒、额外检查

阶段三：规则评估（Rule Evaluation）
  └── deny-first 引擎评估当前操作
  └── 检查：工具类型、内容前缀、用户身份、环境上下文
  └── 输出：允许 / 拒绝 / 需要审批

阶段四：权限处理器（Permission Handler）
  └── 根据评估结果路由到四条路径之一
```

**阶段四的四条处理器路径**：

| 路径 | 触发条件 | 行为 |
|------|---------|------|
| **Coordinator** | 需要用户交互审批 | 弹出权限提示，等待用户确认 |
| **Swarm worker** | 分布式场景 | 将审批请求路由到可用 worker |
| **Speculative classifier** | 自动模式 | 分类器判断 allow/deny/request（见 20.5） |
| **Interactive** | 交互式会话 | 在当前终端显示提示 |

**为什么分四个阶段**：

- **预过滤**在最早阶段移除危险工具，效率最高（模型根本看不到被禁工具）
- **Hook** 提供动态扩展点，让插件和自定义逻辑能介入
- **规则评估** 是核心决策引擎，确保 deny-first 原则
- **处理器** 将决策结果转化为用户可感知的行为（提示/自动执行/拒绝）

**结论**：授权流水线是"分层筛子"——越危险的工具越早被过滤，越接近执行点的检查越精细。

**怎么做**：
- 工具被预过滤移除 → 检查 deny rules 是否过于宽泛
- 工具通过了预过滤但在 hook 阶段被拒绝 → 检查 PreToolUse hooks
- 工具通过了所有阶段但你需要审批 → 这是正常流程，根据上下文决定批准或拒绝

## 20.5 Auto-Mode 分类器

### 问题：自动模式是怎么判断一个操作是否安全的？它会不会误判？

**背景**：`auto` 权限模式是 Claude Code 的高级功能——系统自动判断操作是否安全，无需每次询问用户。但用户会担心："系统会不会误判？把危险操作当成安全的自动执行了？"

**为什么是问题**：自动模式的便利性建立在信任基础上。如果不理解分类器的决策机制，用户要么不敢用（失去便利性），要么盲目信任（失去安全性）。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 5.3）揭示了分类器的设计：

**分类器的输入：三种 prompt resources**

分类器在决策时加载三种 prompt 资源：

1. **Base system prompt**：分类器的基础行为定义
2. **External template**：外部可配置的模板（项目/用户级）
3. **Internal template**：内部模板（系统固定）

**分类器的输出**：

| 输出 | 含义 | 后续行为 |
|------|------|---------|
| **allow** | 操作安全，自动执行 | 工具直接调用，不提示用户 |
| **deny** | 操作危险，拒绝执行 | 模型收到拒绝原因，可修正后重试 |
| **request manual approval** | 不确定，需要人判断 | 弹出权限提示 |

**Recovery-oriented denial（面向恢复的拒绝）**：

这是分类器的关键设计原则。当分类器输出 **deny** 时，它不是简单地"阻止操作结束"，而是：

1. 向模型返回**拒绝原因**（如"此命令包含 rm -rf，可能删除文件"）
2. 模型可以**修正操作**（如改为 `rm` 具体文件）
3. 修正后的操作**重新进入分类器**
4. 如果修正后安全，分类器输出 allow

这意味着 deny 是** routing signal（路由信号）**而非 hard stop（硬停止）。模型有机会学习和调整，而不是被一棍子打死。

**与 deny rules / sandboxing 的关系**：

分类器与 deny rules、沙箱是**独立运作**的三层防御：

```
操作请求
  ↓
Deny Rules（硬规则）→ 匹配 deny → 直接拒绝（不可绕过）
  ↓ 通过
Auto-Mode Classifier（智能判断）→ deny → 模型可修正重试
  ↓ allow
Sandbox（物理限制）→ 即使允许，物理边界仍然生效
  ↓
执行
```

**结论**：自动模式分类器是"智能的判断层"，不是"唯一的防线"。它下面有 deny rules 兜底，上面有 sandbox 收尾。分类器的 deny 是可恢复的——模型可以修正后重试。

**怎么做**：
- 使用 auto 模式时，注意观察分类器的拒绝原因 → 如果频繁拒绝正常操作，说明分类器过于保守
- 如果分类器 allow 了危险操作 → 检查 deny rules 是否遗漏，或切换到更严格的模式
- auto 模式不替代 deny rules → 两者配合使用，deny rules 处理已知危险，分类器处理未知场景

## 20.6 自动模式 vs 其他方案的对比

### 问题：其他 AI 编码工具是怎么做安全的？Claude Code 的方式更好吗？

**背景**：了解其他方案的设计取舍，有助于理解 Claude Code 权限系统的独特之处。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 5.3）对比了三种主流安全策略：

| 方案 | 机制 | 优点 | 缺点 |
|------|------|------|------|
| **Docker 容器隔离（SWE-Agent/OpenHands）** | 在隔离容器中运行所有操作 | 强隔离，即使恶意操作也只影响容器 | 启动慢，资源占用高，文件同步复杂 |
| **Git rollback（Aider）** | 允许操作执行，但保留完整回滚能力 | 无摩擦，用户体验流畅 | 危险操作已经执行，回滚不一定完美 |
| **Claude Code 分层策略** | 预过滤 + deny-first + 分类器 + 可选沙箱 | 分层防御，渐进自治，可配置 | 系统复杂，需要理解各层交互 |

**为什么 Claude Code 不选 Docker 作为默认**：

Docker 隔离确实更强，但：
- 启动延迟秒级（worktree 是毫秒级）
- 文件系统需要同步（容器内修改怎么回到主机？）
- 不适合日常开发的高频交互

Claude Code 的策略是"分层策略执行 + 可选容器沙箱"——默认用轻量级机制（deny rules、分类器），需要强隔离时再用 Docker。

**为什么不用纯 Git rollback**：

Git rollback 假设"任何操作都可以撤销"。但：
- `rm -rf /important-data` 执行后，Git 无法恢复被删除的文件
- 外部 API 调用（发送邮件、删除云资源）无法通过 Git 回滚
- 安全应该在**执行前**阻止，而非**执行后**恢复

**结论**：没有完美的安全方案，只有适合场景的取舍。Claude Code 选择了"多层轻量防御 + 可选强隔离"的混合策略。

**怎么做**：
- 日常开发 → Claude Code 默认分层策略足够
- 需要强隔离的不可信代码 → 启用 Docker 沙箱（第21章）
- 不要依赖"事后回滚"作为安全策略 → 配置适当的 deny rules

## 20.7 安全层独立性与共享故障模式

### 问题：如果一层安全机制失败了，其他层能补上吗？

**背景**：Claude Code 的安全架构有多层（deny rules、hooks、分类器、沙箱）。理论上，一层失败时其他层应该接管。但这是否真的成立？

**为什么是问题**：Defense in Depth（纵深防御）的核心假设是"层间独立"——一层失败不会导致整个系统崩溃。但如果多层共享同一个故障源，这个假设就不成立。

**独立性假设 vs 共享故障风险**：

**理论上独立的层**：
- deny rules（静态规则引擎）
- hooks（动态插件逻辑）
- 分类器（AI 模型判断）
- 沙箱（操作系统级隔离）

**实际的共享故障源**：

*Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems*（Section 11.3）指出了一个关键案例：

> "When >50 subcommands are pending, the system falls back to a single generic approval prompt due to performance constraints."

翻译：当待处理的子命令超过 50 个时，系统因性能限制回退到**单一通用审批提示**。

这意味着：在极端并发场景下，所有精细化的权限判断（deny rules、分类器）被压缩成一个"批准所有/拒绝所有"的二元选择。各层不再独立运作，而是被统一回退机制接管。

**安全 vs 性能的结构性张力**：

| 场景 | 安全需求 | 性能需求 | 权衡结果 |
|------|---------|---------|---------|
| 少量操作 | 精细判断每个操作 | 无压力 | 分层防御正常工作 |
| 大量并发 | 仍然精细判断 | 延迟要求 | >50 时回退到通用审批 |
| 紧急恢复 | 快速执行 | 最低延迟 | bypassPermissions（人工启用） |

**结论**：安全层的独立性是**概率性的**，不是绝对的。在正常负载下各层独立工作，在极端负载下可能回退到简化模式。理解这个限制，才能在高压场景下做出正确决策。

**怎么做**：
- 避免一次性触发 >50 个并发子命令 → 分批处理，保持分层防御的有效性
- 高压场景下，如果收到"通用审批"提示 → 意识到精细化判断已回退，更加谨慎
- 关键操作（生产环境修改）→ 在低负载时执行，确保完整安全栈生效

## 20.8 Fail-Closed 默认值

### 问题：新安装的插件或 MCP 有权限吗？默认安全吗？

**背景**：当你安装一个新插件或连接一个新 MCP 服务器时，它的工具默认处于什么权限状态？如果默认允许所有操作，一个恶意插件就能立刻获得完整权限。

**为什么是问题**：默认权限是安全架构的"基础假设"。如果默认开放，用户必须主动限制；如果默认封闭，用户必须主动授权。后者更安全，但前者更"方便"。

**Fail-Closed 原则**：Claude Code 遵循 **Fail-Closed（故障关闭）** 设计——新工具默认**不安全**，必须经过显式授权才能使用。

**具体表现**：
- 新安装的 MCP 服务器 → 工具默认不在可用池中，需要用户显式连接
- 新插件提供的工具 → 受插件 manifest 中的权限声明限制
- 新子代理类型 → 默认 restricted 权限模式
- 未知工具 → deny-first，默认拒绝

**为什么不是 Fail-Open**：

Fail-Open（故障开放）的系统在故障时保持运行，但可能暴露风险。Claude Code 选择 Fail-Closed，因为：
- Agent 工具的危险性高于传统软件（AI 可能自主调用工具）
- 用户可能不了解新插件的全部能力
- "事后撤销权限"不如"事前授权"安全

**结论**：Fail-Closed 是 Claude Code 安全哲学的底线。任何新能力默认不可访问，直到你显式批准。

**怎么做**：
- 安装新插件/MCP 后 → 审查它提供的工具和权限声明
- 不要批量批准未知工具 → 逐个确认每个工具的用途
- 定期审计已授权的工具列表 → 撤销不再需要的权限

## 20.9 用户行为数据：信任轨迹

### 问题：用户真的接受自动模式吗？权限系统在实际使用中表现如何？

**背景**：安全设计的有效性最终要通过用户行为数据验证。如果用户频繁绕过权限系统，说明设计有缺陷；如果用户在高信任场景中流畅工作，说明设计成功。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 2.1）提供了关键数据：

| 指标 | 数据 | 含义 |
|------|------|------|
| **批准率** | **93%** | 绝大多数权限提示被用户批准，系统没有过度敏感 |
| **信任轨迹** | **~20% → >40%** | 用户从 plan/default 逐渐切换到更自动的模式 |
| **模式切换** | 用户主动切换 | 不是系统强制，是用户基于信任自行选择 |

**信任轨迹的解读**：

- **初期（~20% 自动操作）**：新用户从 plan 或 default 开始，几乎每个操作都需要确认
- **中期（~30% 自动操作）**：用户逐渐熟悉系统，对常规操作（如 `Read`、`Grep`）切换到 acceptEdits 或 auto
- **后期（>40% 自动操作）**：用户在特定场景（如四角色流水线）中完全信任自动模式

**为什么是 40% 而非 100%**：即使经验丰富的用户，也不会把所有操作设为自动。关键操作（`Bash rm`、`Write` 到生产配置）仍然保留人工审批。40% 表示"大量常规操作自动，关键操作人工"。

**结论**：用户行为数据验证了 Claude Code 权限设计的有效性。93% 的批准率说明系统成功区分了"常规操作"和"需要关注的操作"；信任轨迹说明用户在学习和适应后，能做出合理的权限模式选择。

**怎么做**：
- 作为新用户 → 从 default 开始，逐步探索更自动的模式
- 观察你的批准率 → 如果某个操作被频繁提示，考虑调整权限模式或规则
- 关键操作（生产环境、破坏性命令）→ 无论经验如何，保留人工审批
